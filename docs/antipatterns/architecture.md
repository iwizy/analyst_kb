# Анти-паттерны в архитектуре
Страница описывает архитектурные анти-паттерны, которые чаще всего возникают при росте системы и команды.

## Частые анти-паттерны

| Анти-паттерн | Симптом | Последствие | Что делать |
| --- | --- | --- | --- |
| Distributed monolith | множество сервисов, но жесткие синхронные зависимости | сложность как у микросервисов без их плюсов | сокращать chatty calls, вводить bounded contexts и async where possible |
| Преждевременные микросервисы | высокая операционная сложность на маленьком продукте | замедление разработки | стартовать с модульного монолита и эволюционировать |
| God service | один сервис знает «всё» | высокий change coupling и частые инциденты | декомпозиция по бизнес-возможностям |
| Shared database coupling | сервисы разделяют таблицы/схемы | блокировка независимых релизов | database per service или явный integration contract |
| Нет observability by design | диагностика «вслепую» | долгий [MTTR](../glossary.md#abbr-051) | стандарты логирования/метрик/trace с первого релиза |
| Неформализованные архитектурные решения | «устные» договоренности | повтор ошибок и споры | фиксировать решения в [ADR](../glossary.md#abbr-003) |
| Игнорирование [NFR](../glossary.md#abbr-053) | архитектура ориентирована только на FR | деградация [SLA](../glossary.md#abbr-079), рост стоимости | ставить NFR как архитектурные драйверы |
| Нет плана миграции | изменения делаются «большим взрывом» | высокий риск outage | Strangler-подход, phased rollout, rollback-plan |

## Мини-кейс: «микросервисы без платформы»

| Что наблюдали | Корректирующее действие |
| --- | --- |
| каждый сервис имел свои стандарты логирования | ввели единый observability contract |
| API версии не контролировались | ввели semver + API governance |
| частые инциденты при релизах | внедрили canary + automated rollback |

## Сигналы, что архитектура начинает «ломаться»

- рост количества межсервисных вызовов на один пользовательский сценарий;
- рост времени согласования изменений между командами;
- повторяющиеся инциденты в одних и тех же архитектурных узлах;
- архитектурные решения не документированы и зависят от «памяти людей».

## Быстрый план стабилизации (1-2 месяца)

1. Зафиксировать 10-15 ключевых ADR по текущей архитектуре.
2. Провести архитектурный review по топ-рискам.
3. Внедрить базовые reliability patterns в критичных цепочках.
4. Запустить дорожную карту эволюции: `as-is -> target -> migration steps`.
