# Диаграмма Исикавы

Диаграмма Исикавы (fishbone, cause-and-effect) структурирует причины проблемы и помогает перейти от симптомов к корневым факторам. Применяется в разборе инцидентов, дефектов и системных отклонений KPI.

## Цель и задачи

- собрать и структурировать гипотезы причин;
- отделить факты от предположений;
- сформировать проверяемый план корректирующих действий.

## Основные элементы

| Элемент | Назначение |
| --- | --- |
| Эффект (голова) | проблема, которую объясняем |
| Главные "кости" | категории причин (например, 6M) |
| Подпричины | конкретные факторы внутри категории |
| Доказательства | данные, подтверждающие или опровергающие гипотезу |

## Когда применять

- повторяющиеся инциденты без очевидной первопричины;
- много конфликтующих мнений о причинах;
- ретроспективы и postmortem по критическим сбоям.

## Пример

```kroki-plantuml
@startuml
left to right direction
rectangle "Эффект:\nРост ошибок оформления заказа" as Effect
rectangle "Люди" as People
rectangle "Процесс" as Process
rectangle "Системы" as Systems
rectangle "Данные" as Data
rectangle "Метрики" as Metrics

People --> Effect
Process --> Effect
Systems --> Effect
Data --> Effect
Metrics --> Effect

rectangle "Недостаточное обучение" as P1
rectangle "Нет чек-листа" as P2
P1 --> People
P2 --> People

rectangle "Пропущена валидация адреса" as Pr1
rectangle "Нет двойной проверки" as Pr2
Pr1 --> Process
Pr2 --> Process

rectangle "Высокая задержка API" as S1
rectangle "Редкие таймауты" as S2
S1 --> Systems
S2 --> Systems

rectangle "Неполные справочники" as D1
D1 --> Data

rectangle "Нет KPI по ошибкам" as M1
M1 --> Metrics
@enduml
```

## Пошаговый алгоритм построения

1. Сформулируйте проблему измеримо (метрика + период).
1. Выберите категории причин (`6M`, `4P` или кастом под домен).
1. Соберите причины первого уровня на воркшопе.
1. Для ключевых ветвей примените `5 Why` до проверяемой гипотезы.
1. Назначьте владельцев и способ проверки для каждой критичной причины.

## Типовые ошибки

- причины формулируются как решения (`внедрить X`) вместо факторов;
- нет фактов/метрик для проверки гипотез;
- анализ заканчивается диаграммой без плана действий.

## Визуальные рекомендации

- эффект формулируйте одной короткой фразой;
- используйте одинаковую глубину декомпозиции для главных категорий;
- отдельно маркируйте подтвержденные и неподтвержденные причины.

## Связь с другими нотациями

- `Decision Tree`: после выявления причин помогает выбрать корректирующее решение.
- `Impact Map`: связывает причины с инициативами и ожидаемым эффектом.
- `BPMN`: изменения процесса после RCA фиксируются в процессной модели.
