# [Диаграмма](../glossary.md#term-022) Исикавы

Диаграмма Исикавы (Fishbone, Cause-and-Effect) помогает структурировать причины проблемы и перейти от симптомов к корневым факторам.

## Уровни сложности

### Базовый уровень

- формулировать проблему в "голове" диаграммы;
- группировать причины по категориям;
- отличать факт от гипотезы.

### Средний уровень

- проводить сессию причинного анализа с командой;
- приоритизировать причины по влиянию и подтвержденности;
- связывать причины с измеримыми метриками.

### Продвинутый уровень

- интегрировать Fishbone в [RCA](../glossary.md#abbr-070) и проблем-менеджмент;
- комбинировать с 5 Why, Pareto и experiment design;
- выстраивать регулярный цикл улучшений процесса.

## Структура диаграммы

| Элемент | Назначение |
| --- | --- |
| Голова | формулировка проблемы |
| Крупные кости | категории причин |
| Малые кости | конкретные причины |
| Подпричины | уточнение причинных цепочек |

## Типовые категории причин

- People (люди)
- Process (процессы)
- Technology (технологии)
- Data (данные)
- Policy/Management (управление)
- Environment (внешние факторы)

## Пример: низкий [NPS](../glossary.md#abbr-054)

```kroki-plantuml
@startuml
left to right direction
rectangle "Проблема: Низкий NPS" as P
rectangle "Люди\nдолгое обучение поддержки" as C1
rectangle "Процессы\nдолгий SLA ответа" as C2
rectangle "Технологии\nчастые ошибки в checkout" as C3
rectangle "Данные\nнет сегментации жалоб" as C4
rectangle "Управление\nнет owner метрики NPS" as C5

C1 --> P
C2 --> P
C3 --> P
C4 --> P
C5 --> P
@enduml
```

## Плюсы и минусы

| Плюсы | Минусы |
| --- | --- |
| быстро структурирует причины | не доказывает причинность сам по себе |
| вовлекает кросс-функциональную команду | риск перечислить "все подряд" |
| хорошо подходит для ретро и [RCA](../glossary.md#abbr-070) | требует последующей проверки гипотез |

## Когда выбирать

- есть симптом (падение метрики, рост инцидентов), но причина неочевидна;
- нужен фасилитированный разбор с несколькими командами;
- нужно сформировать гипотезы для проверки.

## Как строить

1. Сформулируйте проблему в измеримом виде.
2. Выберите 4-6 категорий причин.
3. Соберите причины и подпричины на сессии.
4. Отметьте подтвержденные и гипотетические причины.
5. Назначьте проверки и владельцев действий.

## Типичные ошибки и как избежать

| Ошибка | Как избежать |
| --- | --- |
| проблема сформулирована абстрактно | используйте конкретную метрику и период |
| смешение причин и решений | причины фиксируйте отдельно от actions |
| отсутствие верификации | каждую приоритетную причину подтверждайте данными |
| нет владельца | назначайте owner и срок проверки |
