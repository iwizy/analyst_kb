# IDEF0

IDEF0 - нотация функционального моделирования, которая описывает, что делает
система и за счет чего это происходит. Основа нотации - модель ICOM:
Input, Control, Output, Mechanism.

## Когда использовать

- нужно разложить сложную функцию на управляемые подфункции;
- важно показать, какие правила управляют функцией;
- требуется согласовать функциональные границы до проектирования решений.

## ICOM-модель

- Input: что преобразуется функцией.
- Control: ограничения, регламенты, правила.
- Output: результат функции.
- Mechanism: исполнители и инструменты.

## Пример: функция "Обработать заявку"

| ICOM | Содержание |
| --- | --- |
| Input | Заявка клиента, приложенные документы |
| Control | Политика KYC, лимиты риска, SLA обработки |
| Output | Решение по заявке, уведомление клиенту |
| Mechanism | Аналитик, CRM, сервис скоринга |

```kroki-plantuml
@startuml
left to right direction

rectangle "A0: Обработать заявку" as A0
rectangle "Input:\nЗаявка клиента\nПриложенные документы" as I
rectangle "Control:\nПолитика KYC\nЛимиты риска\nSLA обработки" as C
rectangle "Output:\nРешение по заявке\nУведомление клиенту" as O
rectangle "Mechanism:\nАналитик\nCRM\nСервис скоринга" as M

I --> A0
C -down-> A0
A0 --> O
M -up-> A0
@enduml
```

## Декомпозиция уровней

- A-0: контекст (что делает система в целом).
- A0: верхнеуровневые функции.
- A1/A2/...: детализация отдельных функций.

Пример декомпозиции A0:

```kroki-plantuml
@startuml
left to right direction

rectangle "A0 Обработать заявку" as A0
rectangle "A1 Проверить\nкомплектность" as A1
rectangle "A2 Оценить риск" as A2
rectangle "A3 Принять решение" as A3
rectangle "A4 Уведомить клиента" as A4

A0 ..> A1 : декомпозиция
A0 ..> A2 : декомпозиция
A0 ..> A3 : декомпозиция
A0 ..> A4 : декомпозиция
@enduml
```

## Практические рекомендации

1. Начинайте с A-0, пока не согласованы границы.
1. На каждом уровне держите 3-6 блоков, иначе модель теряет читаемость.
1. Не подменяйте Control входами и наоборот.
1. Для каждой функции фиксируйте владельца.

## Типичные ошибки

- отсутствие явных управлений (Control);
- слишком глубокая детализация без цели;
- смешение "как делаем" и "что делаем".

## Чек-лист качества

- Для каждой функции заполнены все ICOM-сущности.
- Межуровневая декомпозиция консистентна.
- По модели можно восстановить набор функциональных требований.
