# eEPC

eEPC (extended Event-driven Process Chain) - нотация для моделирования
бизнес-процессов, где ключевая логика строится через чередование событий и
функций. Часто используется в контексте ARIS и регламентов процессов.

## Когда использовать

- нужно описать процесс с акцентом на события и организационную структуру;
- требуется связать функции процесса с ролями, документами и ИТ-системами;
- необходимо подготовить модель для операционного регламента.

## Основные элементы

- Event: состояние/факт ("Заявка получена").
- Function: действие ("Проверить заявку").
- Connectors: AND, OR, XOR.
- Organization Unit/Role: кто выполняет функцию.
- Information Object: документы, данные, формы.

## Пример: согласование договора

```kroki-plantuml
@startuml
start
:Событие:\nЗапрос на договор получен;
:Функция:\nПодготовить проект договора;
if (Требуется правка?) then (Да)
  :Событие:\nТребуется правка;
  :Функция:\nДоработать проект;
  :Событие:\nПроект обновлен;
endif
:Событие:\nПроект готов;
:Функция:\nСогласовать у юриста;
:Событие:\nДоговор согласован;
stop
@enduml
```

Дополнительно на функции указываются:

- роль: юрист, менеджер, руководитель;
- артефакты: шаблон договора, правки, итоговая версия.

## Практический подход

1. Опишите события "до" и "после" каждой функции.
1. Проверяйте правило: две функции подряд без события не допускаются.
1. Привязывайте к функциям роли и документы.
1. Фиксируйте варианты через AND/OR/XOR, а не свободным текстом.

## Типичные ошибки

- превращение eEPC в произвольный flowchart;
- отсутствие ролей и документов;
- слишком детальные технические шаги внутри бизнес-функций.

## Чек-лист качества

- Каждая функция имеет входное и выходное событие.
- Для каждого шага понятен исполнитель.
- Альтернативные пути процесса формализованы коннекторами.
