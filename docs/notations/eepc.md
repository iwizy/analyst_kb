# eEPC

eEPC (extended Event-driven Process Chain) - нотация, где процесс строится как чередование событий и функций. Подходит для регламентных процессов, где важно объяснить, что запускает действие и чем оно заканчивается.

## Цель и задачи

- формализовать процесс в логике "событие -> функция -> событие";
- связать шаги процесса с ролями, документами и ИТ-системами;
- поддержать описание бизнес-процессов в архитектурных фреймворках типа ARIS.

## Основные элементы

| Элемент | Назначение |
| --- | --- |
| Event | Состояние/факт, который фиксируется в процессе |
| Function | Действие, изменяющее состояние |
| Connector (AND/OR/XOR) | Логика разветвлений и объединений |
| Organization Unit / Role | Исполнитель функции |
| Information Object | Документ, запись, данные |
| Application System | Система, выполняющая шаг |

## Когда применять

- описывается регламентный процесс для нескольких ролей;
- нужно показать связь процесса с организационной структурой и документами;
- важна трассировка "событие-результат" в каждом шаге.

## Пример

```kroki-plantuml
@startuml
start
:Событие: Заявка получена;
:Функция: Проверить комплектность;
if (Комплект полный?) then (Да)
  :Событие: Заявка корректна;
  :Функция: Передать на оценку риска;
  :Событие: Риск оценен;
  if (Риск допустим?) then (Да)
    :Функция: Подготовить договор;
    :Событие: Договор готов;
    stop
  else (Нет)
    :Функция: Подготовить отказ;
    :Событие: Отказ сформирован;
    stop
  endif
else (Нет)
  :Событие: Нужны уточнения;
  :Функция: Запросить документы;
  stop
endif
@enduml
```

## Пошаговый алгоритм построения

1. Опишите стартовое и целевое событие процесса.
1. Для каждого шага соблюдайте паттерн "Event -> Function -> Event".
1. Укажите роли/подразделения для каждой функции.
1. Привяжите документы и системы, если они критичны для шага.
1. Формализуйте разветвления через `AND/OR/XOR`.

## Типовые ошибки

- две функции подряд без промежуточного события;
- отсутствие привязки функций к исполнителю;
- смешение процессного уровня с низкоуровневыми техническими шагами.

## Визуальные рекомендации

- события формулируйте как состояние (`Заявка зарегистрирована`), а функции как действие (`Проверить заявку`);
- избегайте длинных цепочек без декомпозиции в подпроцессы;
- выделяйте коннекторы явно, чтобы не терялась логика ветвлений.

## Связь с другими нотациями

- `BPMN`: eEPC можно преобразовать в BPMN для автоматизации.
- `IDEF0`: функции eEPC уточняются через ICOM-модель.
- `DFD`: к функциям eEPC добавляются входные/выходные потоки данных.
