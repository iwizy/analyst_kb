# [UML](../../glossary.md#abbr-092)

UML (Unified Modeling Language) используется для описания структуры и поведения системы. В разделе фокус на наиболее полезных для системного аналитика диаграммах: Class, Sequence, Activity, State Machine.

## Уровни сложности

### Базовый уровень

- понимать назначение каждой UML-диаграммы;
- выбирать нужный тип диаграммы под вопрос;
- читать и объяснять базовые элементы UML.

### Средний уровень

- строить связанный набор UML-диаграмм по одному сценарию;
- поддерживать единые имена сущностей между диаграммами;
- использовать UML для анализа требований и интеграций.

### Продвинутый уровень

- выстраивать стандарты UML на уровне команды;
- использовать UML для impact analysis изменений;
- интегрировать UML-модели в docs-as-code и архитектурные ревью.

## Когда использовать UML

| [Диаграмма](../../glossary.md#term-022) | Ключевой вопрос | Когда лучше всего подходит |
| --- | --- | --- |
| Class | как устроены сущности и их связи | доменная модель, [API](../../glossary.md#abbr-005)/DTO, data contracts |
| Sequence | как идет взаимодействие по шагам | интеграционные сценарии, обработка ошибок |
| Activity | как выполняется процесс и ветвления | бизнес-процессы с правилами и условиями |
| State Machine | как объект меняет состояния | жизненный цикл заявки, заказа, подписки |

```kroki-plantuml
@startuml
left to right direction
rectangle "Class\nСтруктура" as C
rectangle "Sequence\nВзаимодействие" as S
rectangle "Activity\nПоток действий" as A
rectangle "State\nЖизненный цикл" as T
C --> S : уточнение сценариев
A --> S : детализация шагов
T --> A : правила переходов
@enduml
```

## Как связать [UML](../../glossary.md#abbr-092) с другими нотациями

- [[BPMN](../../glossary.md#abbr-009)](../bpmn.md): BPMN описывает межрольный процесс, Activity/Sequence уточняют логику шага.
- [[C4](../../glossary.md#abbr-011)](../c4.md): C4 фиксирует архитектурный контекст, UML деталирует поведение компонента.
- [[ERD](../../glossary.md#abbr-032)](../erd.md): Class-модель может быть источником для логической data model.
- [Требования](../../requirements/index.md): UML помогает уточнять сценарии use-case и acceptance criteria.

## Плюсы и минусы UML

| Плюсы | Минусы |
| --- | --- |
| единый международный стандарт моделирования | для бизнеса часть обозначений может быть сложной |
| покрывает и структуру, и поведение системы | легко переусложнить диаграммы деталями |
| поддерживается большинством инструментов | требует дисциплины по актуализации |

## Единые правила для UML-диаграмм

1. На диаграмме отвечайте на один вопрос, а не на все сразу.
2. Не переносите весь код в диаграмму классов.
3. Для Sequence показывайте только значимые вызовы.
4. В Activity явно обозначайте условия ветвления.
5. В State Machine фиксируйте события переходов и guard conditions.
