# UML

UML (Unified Modeling Language) - базовая нотация для описания поведения и структуры систем. В практике аналитика UML закрывает три задачи: формализация сценариев, моделирование предметной области и проверка логики переходов.

## Когда использовать UML

- нужно убрать неоднозначность в требованиях к поведению системы;
- требуется синхронизировать бизнес-логику, API и данные;
- командам разработки и тестирования нужна единая визуальная модель.

## Какие UML-диаграммы использовать

| Вопрос | Диаграмма | Что фиксирует |
| --- | --- | --- |
| Кто с кем и в каком порядке взаимодействует? | [Sequence](sequence.md) | Последовательность сообщений и варианты исходов |
| Как выглядит поток действий с условиями? | [Activity](activity.md) | Алгоритм, ветвления, параллельные шаги |
| Какие объекты и связи есть в модели? | [Class](class.md) | Статическая структура и кратности |
| Как объект меняет статус по событиям? | [State Machine](state-machine.md) | Жизненный цикл и допустимые переходы |

## Сквозной процесс моделирования

1. Начните с пользовательского сценария или бизнес-процесса.
1. Постройте `Activity` или `Sequence` для поведения.
1. Выделите доменные объекты и отразите их в `Class`.
1. Для объектов со статусами соберите `State Machine`.
1. Проверьте непротиворечивость между диаграммами и требованиями.

## Связь UML с другими нотациями

| UML | Связь |
| --- | --- |
| BPMN | Шаг BPMN можно детализировать `Activity` или `Sequence` |
| C4 | На уровне компонентов C4 полезно уточнять внутреннюю модель через UML |
| ERD | Сущности `Class` синхронизируются с таблицами и связями в ERD |
| DFD | Потоки данных в DFD уточняются сообщениями в `Sequence` |

## Типовые ошибки

- попытка описать всю систему одной диаграммой;
- смешение бизнес-терминов и технических деталей без уровня абстракции;
- отсутствие актуализации диаграмм после изменения требований.
