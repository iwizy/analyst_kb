# Archimate

ArchiMate - нотация для описания архитектуры предприятия. Она связывает бизнес,
приложения, данные и технологическую инфраструктуру в единую модель.

## Когда использовать

- нужно показать, как бизнес-возможность поддерживается ИТ-ландшафтом;
- требуется единая карта зависимостей между процессами, сервисами и системами;
- важно оценить влияние изменений (impact analysis) до запуска инициативы.

## Базовые слои и аспекты

- Business layer: роли, процессы, бизнес-сервисы.
- Application layer: приложения, компоненты, приложения-сервисы.
- Technology layer: узлы, сети, технологические сервисы.
- Motivation/Strategy: драйверы, цели, принципы, результаты.

## Как строить модель

1. Зафиксируйте бизнес-цель изменения.
1. Выберите точку входа: capability, value stream или ключевой процесс.
1. Опишите связанные приложения и данные на уровне компонентов.
1. Добавьте технологические зависимости (платформа, сеть, среда исполнения).
1. Проверьте, что по модели можно ответить на вопрос "что сломается при изменении".

## Пример: цифровое открытие счета

```kroki-plantuml
@startuml
left to right direction
skinparam componentStyle rectangle

rectangle "Цель: сократить время\nоткрытия счета до 15 минут" as GOAL

package "Business Layer" {
  rectangle "Открытие счета\n(Business Process)" as BP
  actor "Клиентский менеджер\n(Business Role)" as BR
}

package "Application Layer" {
  rectangle "KYC Check API\n(Application Service)" as AS
  rectangle "Onboarding Service\n(Application Component)" as AC
}

package "Technology Layer" {
  node "Kubernetes Cluster\n(Technology Node)" as TN
}

GOAL --> BP : поддерживается
BR --> BP : выполняет
BP --> AS : использует
AS --> AC : реализуется
AC --> TN : развернут на
@enduml
```

Что дает такая модель:

- видна связь бизнес-цели с конкретными ИТ-компонентами;
- легко локализовать зоны риска при изменениях API или инфраструктуры.

## Типичные ошибки

- смешивание нескольких уровней детализации в одной диаграмме;
- отсутствие явных связей между целью, процессом и компонентами;
- попытка "нарисовать все сразу" без выбранного viewpoint.

## Чек-лист качества

- Выбран viewpoint под конкретный вопрос.
- Указаны границы системы и внешние участники.
- Модель читается слева направо или сверху вниз без пересечений.
- Понятно, какие элементы критичны для цели.
