# Impact Map

Impact Map связывает цель продукта с участниками, их поведением и конкретными deliverables, чтобы избежать реализации "фич ради фич".

## Уровни сложности

### Базовый уровень

- формулировать бизнес-цель (Why);
- определять акторов (Who);
- описывать желаемое изменение поведения (How);
- фиксировать deliverables (What).

### Средний уровень

- использовать impact map в discovery и приоритизации;
- добавлять гипотезы и критерии успеха;
- отсекать низкоценные инициативы.

### Продвинутый уровень

- масштабировать подход на product portfolio;
- связывать impact map с experiment backlog и A/B тестами;
- управлять инвестициями по expected impact.

## Элементы Impact Mapping

| Вопрос | Элемент |
| --- | --- |
| Why | цель бизнеса |
| Who | актеры, влияющие на цель |
| How | изменение поведения актера |
| What | функциональность/инициативы |

## Пример: запуск нового продукта подписки

```kroki-plantuml
@startuml
left to right direction
rectangle "Why\nУвеличить MRR на 20 pct" as WHY
rectangle "Who\nНовые пользователи" as WHO1
rectangle "Who\nТекущие клиенты" as WHO2
rectangle "How\nПовысить конверсию trial->paid" as HOW1
rectangle "How\nСнизить churn" as HOW2
rectangle "What\nОнбординг + paywall A/B" as WHAT1
rectangle "What\nПакеты лояльности" as WHAT2

WHY --> WHO1
WHO1 --> HOW1
HOW1 --> WHAT1
WHY --> WHO2
WHO2 --> HOW2
HOW2 --> WHAT2
@enduml
```

## Плюсы и минусы

| Плюсы | Минусы |
| --- | --- |
| удерживает фокус на бизнес-эффекте | требует зрелой продуктовой аналитики |
| снижает риск лишней разработки | может выглядеть абстрактно без метрик |
| улучшает коммуникацию discovery-delivery | нуждается в регулярном пересмотре |

## Когда выбирать

- нужно объяснить, зачем делается функциональность;
- требуется приоритизация discovery backlog;
- нужно синхронизировать продукт, аналитику и разработку.

## Как строить

1. Зафиксируйте измеримую цель `Why`.
2. Выделите ключевых акторов `Who`.
3. Опишите поведенческие изменения `How`.
4. Предложите deliverables `What` как гипотезы.
5. Свяжите с метриками и планом экспериментов.

## Типичные ошибки и как избежать

| Ошибка | Как избежать |
| --- | --- |
| `What` без `How` | всегда доказывайте причинную связь |
| цель без метрики | формулируйте KPI заранее |
| слишком много акторов | оставляйте тех, кто реально влияет |
| отсутствие проверки гипотез | планируйте эксперимент и критерии stop/go |

## Связь с требованиями и roadmap

- `What` переводится в [эпики и требования](../requirements/specification-approaches.md);
- `How` помогает обосновать [приоритизацию](../requirements/management/prioritization.md).

## Инструменты и форматы

- Impact Mapping board в Miro/FigJam, draw.io, mind-map tools;
- экспорт: SVG/PNG/PDF.

## Контрольные вопросы

1. Цель impact map измерима и привязана ко времени?
2. Для каждого deliverable есть гипотеза влияния?
3. Понятно ли, какие метрики подтвердят успех?
4. Какие инициативы не имеют доказанного impact?

## Чек-лист самопроверки

- заполнены Why-Who-How-What;
- есть KPI и baseline;
- гипотезы приоритизированы;
- связь с требованиями и roadmap прозрачна;
- назначены владельцы экспериментов.
