# Нотации, диаграммы, моделирование

Раздел помогает выбрать правильный способ визуализации: как описать процессы,
архитектуру, данные, причины проблем и стратегические цели так, чтобы команда
понимала модель одинаково.

## Что дает системному аналитику

- быстрый переход от текстового требования к наглядной модели;
- снижение неоднозначности на согласованиях;
- общий язык между бизнесом, разработкой, тестированием и архитектурой;
- основу для трассировки решений и изменений.

## Как выбрать нотацию под задачу

| Задача | Нотация | Практический результат |
| --- | --- | --- |
| Показать поток работ с ролями и развилками | [BPMN](bpmn.md) | Модель бизнес-процесса для согласования и автоматизации |
| Описать поток данных между системами | [DFD](dfd.md) | Понимание интеграций, входов и выходов |
| Разложить функцию на уровни управления | [IDEF0](idef0.md) | Структурная функциональная декомпозиция |
| Смоделировать архитектуру на нескольких уровнях | [C4](c4.md), [UML](uml/index.md), [Archimate](archimate.md) | Архитектурные представления для разных ролей |
| Спроектировать модель хранения данных | [ERD](erd.md) | Каркас структуры БД и связей |
| Разобрать причины проблемы | [Диаграмма Исикавы](fishbone-diagram.md) | Карта причин и гипотез для устранения дефекта |
| Сформировать стратегию изменений продукта | [Impact Map](impact-map.md), [Карта целей](goals-map.md) | Связь целей, метрик, инициатив и эффектов |
| Оценить управленческую зрелость практик | [Модели зрелости](maturity-models/index.md) | План развития процессов по уровням зрелости |

## Минимальный стандарт качества диаграмм

- У каждой схемы есть цель, аудитория и владелец.
- Понятен уровень детализации: от контекста к деталям.
- Имена объектов однозначны и говорят о сути.
- На диаграмме отражены границы и внешние зависимости.
- Есть дата актуализации и связь с требованиями/решениями.
