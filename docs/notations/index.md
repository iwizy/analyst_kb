# Нотации и моделирование

Раздел описывает рабочий набор нотаций системного аналитика: от бизнес-процессов и архитектуры до данных, причинно-следственного анализа и стратегических карт.

## Как использовать раздел

1. Определите вопрос, на который должна отвечать модель.
1. Выберите нотацию под этот вопрос, а не по привычке.
1. Согласуйте уровень детализации с аудиторией.
1. Зафиксируйте связи с требованиями, API, архитектурными решениями и тестами.

## Связи между нотациями

| Откуда | Куда | Как связать |
| --- | --- | --- |
| BPMN | UML Activity | Детальные алгоритмы отдельных шагов процесса |
| BPMN | UML Sequence | Технические вызовы между сервисами внутри шага BPMN |
| DFD | API/интеграции | Потоки данных в DFD превращаются в контракты API и события |
| C4 Container | UML Class/ERD | Уточнение внутренней модели данных и объектов контейнера |
| ERD | UML Class | Сущности и связи синхронизируются с доменной моделью |
| Карта целей | Impact Map | Цель и KPI раскладываются на акторов, поведение и поставки |
| Impact Map | BPMN/Backlog | Гипотезы поставок переводятся в процессы, user stories и задачи |
| Модели зрелости | Все разделы | Выявляют, какие практики моделирования нужно развивать |

```kroki-plantuml
@startuml
left to right direction
rectangle "Цели и изменения" as G
rectangle "Процессы" as P
rectangle "Архитектура" as A
rectangle "Данные" as D
rectangle "Правила" as R
rectangle "Зрелость" as M

G --> P : инициативы
G --> A : целевая архитектура
P --> A : требования к сервисам
A --> D : модель хранения
P --> R : правила решений
M --> G : приоритеты развития
M --> P : стандарты процессов
M --> A : архитектурное управление
M --> D : качество данных
@enduml
```

## Минимальный стандарт качества любой диаграммы

- У диаграммы есть цель, аудитория и владелец.
- Указан уровень детализации: контекст, логика, реализация.
- Названия элементов соответствуют глоссарию проекта.
- Отражены границы системы и внешние зависимости.
- Есть дата актуализации и ссылка на связанный артефакт (требование, ADR, тикет).
