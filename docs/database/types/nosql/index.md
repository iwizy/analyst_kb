# NoSQL базы данных

NoSQL объединяет модели хранения, оптимизированные под конкретные паттерны нагрузки: гибкая схема, высокий write throughput, масштабирование и специализированные запросы.

## Когда выбирать NoSQL

- схема данных часто меняется;
- нагрузка распределенная и write-heavy;
- критична масштабируемость по горизонтали;
- часть данных допускает eventual consistency;
- доминирует конкретный тип операций (поиск, граф, time series, key-value).

## Модели NoSQL

| Модель | Ключевая идея | Типовые задачи |
| --- | --- | --- |
| Документо-ориентированные | хранение JSON-документов | профили, каталоги, CMS, API payload |
| Key-Value | быстрый доступ по ключу | кэш, сессии, rate limiting |
| Wide-column | строки с динамическими колонками | телеметрия, event store, high write |
| Time-series | оптимизация по времени | метрики, IoT, observability |
| Графовые | узлы и связи | рекомендации, fraud graph, knowledge graph |
| Поисковые | полнотекстовые индексы | поиск, лог-аналитика, relevance ranking |

## Практический подход

1. Выделите критичный паттерн чтения/записи.
1. Определите допустимую задержку консистентности.
1. Спроектируйте ключи и партиционирование до разработки.
1. Проверьте модель на сценариях масштабирования и отказов.

## Типичные ошибки

- выбирать NoSQL "на будущее" без конкретного паттерна;
- переносить реляционную модель без адаптации;
- игнорировать стратегию ключей и TTL;
- не планировать reindex/repartition и миграции.
