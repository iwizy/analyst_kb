# Шаблон выбора стиля архитектуры

Шаблон применяется при выборе между монолитом, модульным монолитом и микросервисами.

## Входные сигналы

| Параметр | Низкое значение | Высокое значение |
| --- | --- | --- |
| Размер команды | 1-2 команды | 5+ команд |
| Частота релизов | 1 раз в месяц | несколько раз в день |
| Разнородность нагрузки | близкая | сильно разная по доменам |
| Автономия доменов | низкая | высокая |
| Зрелость платформы | базовая | зрелая [CI/CD](../glossary.md#abbr-017), observability |

## Матрица решения

| Критерий | Монолит | Модульный монолит | Микросервисы |
| --- | --- | --- | --- |
| Скорость старта | высокая | средняя | низкая |
| Операционная сложность | низкая | средняя | высокая |
| Независимый релиз частей | низкий | средний | высокий |
| Локализация отказов | низкая | средняя | высокая |
| Цена изменений на масштабе | растет быстро | контролируемо | зависит от зрелости платформы |

## Рекомендация по умолчанию

- Для нового продукта с ограниченной командой: **модульный монолит**.
- Для MVP с жестким дедлайном: **монолит** с планом модульной эволюции.
- Для большого продукта с автономными доменами и зрелой платформой: **микросервисы**.

## Шаблон фиксации решения

```markdown
Контекст:
- Команды:
- Частота релизов:
- NFR:

Решение:
- Выбранный стиль:
- Границы применимости:

Эволюционный план:
- Этап 1:
- Этап 2:
- Этап 3:
```

## Пример (e-commerce)

- Контекст: 3 продуктовые команды, weekly релизы, пиковая сезонная нагрузка.
- Решение: модульный монолит с жесткими доменными границами и отдельными read-моделями.
- План: через 2 квартала выделить каталог и checkout в отдельные сервисы.

## Частые риски и меры

| Риск | Что сделать заранее |
| --- | --- |
| Распределенный монолит | определить API-границы и запретить cross-domain доступ к БД |
| Преждевременные микросервисы | запускать только после внедрения platform practices |
| Монолит без модульности | внедрить архитектурные тесты и ownership по модулям |
