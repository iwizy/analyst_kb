# Шаблон выбора API-протокола

Шаблон помогает выбирать между REST, GraphQL, gRPC и SOAP на основании контекста и ограничений.

## Сравнение вариантов

| Критерий | REST | GraphQL | gRPC | SOAP |
| --- | --- | --- | --- | --- |
| Публичные API для партнеров | сильный | средний | слабый | нишевый |
| Производительность внутреннего RPC | средний | средний | сильный | слабый |
| Гибкость выборки данных | средний | сильный | средний | слабый |
| Строгая контрактность | средний | средний | сильный | сильный |
| Порог входа экосистемы | низкий | средний | средний | высокий |

## Рекомендация по умолчанию

- Внешний API: REST + OpenAPI.
- Внутренний high-throughput вызов: gRPC.
- UI-агрегация и гибкая выборка: GraphQL (при зрелом governance).
- Legacy/регуляторный B2B: SOAP.

## Шаблон решения

```markdown
Контекст:
- Тип клиентов: public/private/internal
- Трафик: RPS, payload
- Требования к совместимости:

Выбор:
- Основной протокол:
- Допустимые исключения:
- Политика versioning:
- Политика backward compatibility:
```

## Пример (gov API)

- Контекст: внешние интеграторы, юридическая значимость и долгий срок поддержки.
- Решение: REST + строгая версия в URL + контрактные тесты.
- Исключение: SOAP-шлюз для legacy-партнеров в период миграции.

## Частые ошибки

- смешивать несколько стилей в одном API без правил;
- менять контракт без стратегии деприкации;
- выбирать gRPC для внешних партнеров без зрелого DX.
