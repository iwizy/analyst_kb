# Шаблон выбора формата требований

Шаблон помогает выбрать формат фиксации требований под контекст проекта.

## Сравнение форматов

| Формат | Лучший контекст | Плюсы | Ограничения |
| --- | --- | --- | --- |
| User Story | Agile delivery, продуктовые команды | быстрый цикл, фокус на ценности | риск потери системного контекста |
| Use Case | сложные пользовательские сценарии | хорошо раскрывает потоки и исключения | сложнее поддерживать при частых изменениях |
| SRS/BRD | регуляторные и контрактные проекты | полнота и аудируемость | ниже скорость изменений |
| BDD/SBE | высокие риски неоднозначности | единое понимание поведения | требует дисциплины примеров |

## Рекомендация по умолчанию

- Product/Agile: User Story + Acceptance Criteria.
- Сложная логика процесса: User Story + Use Case для критичных сценариев.
- Регуляторный контур: SRS/BRD как baseline, delivery через backlog.

## Шаблон решения

```markdown
Контекст:
- Методология (Agile/Waterfall/Hybrid):
- Требования регулятора/контракта:
- Частота изменений:

Формат:
- Основной:
- Дополнительный:
- Правила версии и baseline:
```

## Пример (госуслуги)

- Контекст: высокий уровень регуляторики и аудит.
- Решение: SRS как юридически значимая база + user stories для итеративной разработки.
- Контроль: трассировка SRS -> backlog -> test cases через [RTM](../glossary.md#abbr-107).
