# Шаблон выбора хранилища данных

Шаблон помогает принять решение между реляционной БД, NoSQL и NewSQL.

## Ключевые входные параметры

| Параметр | Что уточнить |
| --- | --- |
| Инварианты и транзакционность | нужна ли строгая [ACID](../glossary.md#abbr-096)-модель |
| Профиль нагрузки | read/write ratio, пики, размер данных |
| Тип запросов | сложные join/aggregation или простые key-access |
| Эволюция схемы | жесткая/стабильная или часто изменяемая |
| Масштабирование | single-region или multi-region |

## Матрица решения

| Условие | SQL | NoSQL | NewSQL |
| --- | --- | --- | --- |
| Строгие транзакции и связи | сильный выбор | ограниченно | сильный выбор |
| Быстрая эволюция структуры | средне | сильно | средне |
| Глобальная горизонтальная масштабируемость | средне | сильно | сильно |
| Сложная аналитика в том же контуре | сильный выбор | зависит от движка | средне |

## Рекомендация по умолчанию

- Primary store для критичных бизнес-операций: SQL.
- Специализированные контуры (поиск, документы, time-series): NoSQL рядом с primary.
- NewSQL выбирать при сильной потребности в масштабировании и ACID одновременно.

```markdown
Контекст:

Выбор:
- Primary store:
- Secondary stores:

```

## Пример (fintech core)

- Primary: PostgreSQL для счетов и проводок.
- Secondary: Redis для hot reads, Kafka + ClickHouse для аналитики.
- Обоснование: строгая целостность в core + масштабируемая аналитика в отдельном контуре.
