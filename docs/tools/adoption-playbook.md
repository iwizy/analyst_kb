# Playbook внедрения инструментов
Страница описывает, как внедрять инструменты системного аналитика поэтапно: от выбора до масштабирования на несколько команд.

## Уровни сложности

### Базовый уровень

- выбрать инструмент под конкретный артефакт;
- провести короткий пилот и собрать обратную связь;
- зафиксировать минимальные правила использования.

### Средний уровень

- собрать единый toolchain и интеграции между инструментами;
- внедрить шаблоны и governance-политики;
- измерять эффект внедрения на скорости и качестве.

### Продвинутый уровень

- масштабировать стек на программу/портфель;
- управлять [TCO](../glossary.md#abbr-087) и рисками vendor lock-in;
- строить roadmap развития инструментальной платформы.

## Этапы внедрения

| Этап | Цель | Результат |
| --- | --- | --- |
| Discovery | определить проблемы процесса | список целевых use-case |
| Pilot | проверить гипотезу на 1 команде | решение go/no-go |
| Rollout | развернуть на нескольких командах | стандартизированный workflow |
| Scale | оптимизировать и автоматизировать | стабильный platform operating model |

## Минимальный пилотный план (4-6 недель)

1. Выбрать 1-2 критичных сценария (например, требования + API контракт).
2. Определить baseline-метрики до внедрения.
3. Настроить интеграции (Git/[CI](../glossary.md#abbr-016), Jira/Confluence).
4. Провести обучение команды.
5. Сравнить метрики до/после и принять решение.

## Матрица внедрения по классам инструментов

| Класс | Quick win | Риск | Контроль |
| --- | --- | --- | --- |
| Моделирование | единые шаблоны диаграмм | расхождение нотаций | naming conventions + review |
| Требования | единая трассировка | дублирование артефактов | single source of truth |
| API | contract-first + автопроверки | breaking changes | compatibility checks в CI |
| БД | управляемые миграции | schema drift | migration policy + rollback |
| Интеграции | мониторинг и alerting | скрытые сбои цепочек | end-to-end observability |

## Шаблон решения о внедрении

| Поле | Пример |
| --- | --- |
| Проблема | низкая связность требований и тестов |
| Цель | повысить traceability до 95% |
| Инструмент | Jira + Xray + Confluence |
| Пилот | 1 команда, 2 спринта |
| KPI | lead time, defect leakage, user adoption |
| Решение | rollout на 4 команды |

## Метрики эффективности

| Группа метрик | Примеры |
| --- | --- |
| Adoption | % активных пользователей, completion rate обучения |
| Delivery | lead time, cycle time, release predictability |
| Quality | defect leakage, contract violation count |
| Economics | стоимость лицензий и поддержки, [ROI](../glossary.md#abbr-073) |

## Типовые ошибки

- внедрение сразу на все команды без пилота;
- отсутствие владельца инструмента;
- отсутствие шаблонов и правил именования;
- фокус на лицензии без оценки операционных затрат.
