# Ситуационный playbook по софт-скилам
Страница содержит прикладные сценарии из работы системного аналитика и готовые тактики коммуникации, переговоров, фасилитации и урегулирования конфликтов.

## Уровни сложности

### Базовый уровень

- распознавать тип рабочей ситуации и выбирать базовую технику;
- использовать короткие шаблоны вопросов и фиксации договоренностей;
- снижать риск недопонимания в коммуникации.

### Средний уровень

- сочетать несколько техник в одной встрече;
- управлять сложными стейкхолдерами;
- сохранять фокус на фактах и целях при конфликте.

### Продвинутый уровень

- фасилитировать multi-team встречи с конфликтом интересов;
- проводить сложные переговоры по scope/срокам/рискам;
- формировать культуру конструктивной обратной связи.

## Сценарий 1. Конфликт по приоритету требований

### Контекст

Бизнес требует срочный функционал, команда разработки настаивает на закрытии технического долга.

### Тактика

1. Зафиксировать интересы сторон (не позиции).
2. Разделить «обязательно сейчас» и «можно отложить».
3. Применить матрицу impact/effort.
4. Согласовать компромисс и дату пересмотра.

### Рабочий шаблон фразы

`Давайте зафиксируем общий критерий успеха на ближайший релиз и на его основе договоримся о балансе между новой ценностью и устойчивостью системы.`

## Сценарий 2. Переговоры о сроке релиза

### Контекст

Стейкхолдер ожидает фиксированную дату, но объем требований меняется.

### Тактика

- применить BATNA/WATNA для прозрачности рисков;
- предложить 2-3 релизных сценария;
- зафиксировать допущения в письменном виде.

| Сценарий | Scope | Дата | Риск |
| --- | --- | --- | --- |
| A | полный | позднее | высокий риск срыва |
| B | MVP | раньше | управляемый |
| C | phased rollout | поэтапно | средний |

## Сценарий 3. Непродуктивная встреча без решений

### Симптомы

- длинное обсуждение без фиксации outcome;
- участники возвращаются к уже закрытым вопросам;
- нет владельцев последующих действий.

### Тактика фасилитации

1. Перезапустить встречу с четкой `agenda`.
2. Ограничить таймбокс по блокам.
3. Использовать dot voting для спорных пунктов.
4. Зафиксировать `decision + owner + due date`.

## Сценарий 4. Эскалация напряжения в команде

### Контекст

Команда в стрессовом режиме перед релизом, растет токсичность в коммуникации.

### Тактика

- краткая синхронизация по статусу рисков;
- правила коммуникации на период релиза;
- выделение канала для критичных вопросов;
- post-release ретроспектива с action items.

## Сценарий 5. Сложная обратная связь

### Модель

SBI (`Situation -> Behaviour -> Impact`) без обвинений и интерпретаций.

### Пример

`На встрече по согласованию требований (Situation) ты несколько раз перебивал коллег (Behaviour), из-за этого мы не зафиксировали общее решение и ушли в повторное обсуждение (Impact).`

## Универсальный шаблон фиксации договоренностей

| Поле | Что записать |
| --- | --- |
| Решение | что согласовано |
| Ограничения | что остается вне scope |
| Владелец | кто отвечает за выполнение |
| Срок | когда проверяем выполнение |
| Риск | что может помешать |

## Типичные ошибки

- обсуждение позиций вместо интересов;
- отсутствие письменной фиксации договоренностей;
- уход в эмоции без возврата к критериям решения;
- попытка решить все конфликты в одном разговоре.
