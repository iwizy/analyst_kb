# Практический playbook выбора методологии
Страница дает пошаговый алгоритм выбора и адаптации методологии под контекст проекта, чтобы снизить риск «процесс ради процесса».

## Уровни сложности

### Базовый уровень

- выбрать класс подхода: predictive, adaptive, hybrid;
- согласовать роли, артефакты и ритмы;
- зафиксировать критерии успешности процесса.

### Средний уровень

- адаптировать методологию под домен и команду;
- настроить точки контроля качества и рисков;
- увязать методологию с управлением требованиями и релизами.

### Продвинутый уровень

- проектировать модель для multi-team программ;
- планировать переходы между подходами по мере роста продукта;
- управлять зрелостью процесса по метрикам.

## Алгоритм выбора методологии

1. Оцените стабильность требований и частоту изменений.
2. Оцените регуляторные ограничения и требования аудита.
3. Уточните размер команды и степень распределенности.
4. Зафиксируйте целевой time-to-market и приемлемый риск.
5. Сформируйте shortlist из 2-3 подходов.
6. Выберите подход и пилотный период (4-8 недель).

## Матрица «контекст -> методология»

| Контекст | Предпочтительный подход |
| --- | --- |
| стабильные требования, строгий аудит | Waterfall/V-model |
| высокая неопределенность и быстрые итерации | Scrum/Kanban/XP |
| крупная программа с несколькими командами | SAFe/LeSS/Nexus |
| смешанный контур (регуляторика + быстрый delivery) | Water-Scrum-Fall или Stage-Gate + Kanban |

## Сценарии внедрения

### Сценарий 1. Стартап B2C

- старт с Scrum;
- короткий цикл discovery -> delivery;
- акцент на product metrics и быстрый rollback.

### Сценарий 2. Enterprise fintech

- hybrid-модель: формальный upstream + итеративный delivery;
- обязательные архитектурные и риск-review;
- усиленная трассировка требований и тестов.

### Сценарий 3. Гос‑программа

- predictive-гейт для согласований;
- итеративная реализация функциональных блоков;
- формализованные приемки и change governance.

## План перехода между методологиями

| Этап | Цель |
| --- | --- |
| Диагностика | выявить bottleneck текущего процесса |
| Пилот | проверить новый подход на ограниченном scope |
| Масштабирование | распространить успешные практики на остальные команды |
| Стабилизация | зафиксировать стандарты, шаблоны, метрики |

## Мини-шаблон процессного решения

| Поле | Пример |
| --- | --- |
| Контекст | 3 команды, высокий регуляторный контроль |
| Выбранный подход | Water-Scrum-Fall |
| Ключевые ритмы | weekly sync, sprint review, monthly governance |
| Артефакты | BRD/SRS + backlog + release plan |
| Метрики | lead time, defect escape rate, predictability |
| Переоценка | каждые 2 квартала |

## Типичные ошибки

- выбор методологии «по бренду», а не по ограничениям;
- перенос фреймворка без адаптации к организационной культуре;
- отсутствие критериев, по которым процесс считается успешным;
- смешение ролей без явной ответственности.
