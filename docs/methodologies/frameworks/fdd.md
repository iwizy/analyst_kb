# FDD

FDD (Feature-Driven Development) - методика, где планирование и реализация строятся вокруг пользовательских фич.

## Принципы

- feature-centric планирование;
- короткие итерации по фичам;
- явное архитектурное руководство;
- статус по бизнес-функциям, а не по техзадачам.

## Этапы

1. Разработать общую модель.
1. Сформировать список фич.
1. Спланировать по фичам.
1. Спроектировать по фичам.
1. Построить по фичам.

## Артефакты

- feature list;
- feature plan;
- design package;
- build status dashboard.

## Роли

- Chief Architect;
- Chief Programmer;
- Class Owner;
- Domain Expert.

## Плюсы и минусы

Плюсы:

- прозрачный прогресс в терминах ценности;
- хорошо масштабируется по фич-командам.

Минусы:

- высокая зависимость от архитектурного лидерства;
- может быть избыточным для маленьких команд.

## Пример подходящего проекта

Большой B2B-продукт, где релизы организуются по функциональным пакетам для разных клиентских сегментов.

## Связь с требованиями и рисками

- требования группируются в feature sets;
- риск-оценка выполняется на уровне фич;
- отслеживается готовность фичи end-to-end.
