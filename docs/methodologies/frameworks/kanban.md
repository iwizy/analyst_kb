# Kanban

Kanban - потоковый фреймворк управления работой через визуализацию, [WIP](../../glossary.md#abbr-094)-лимиты и pull-систему.

## Уровни сложности

### Базовый уровень

- понимать доску, стадии потока и WIP-лимиты;
- различать lead time и cycle time;
- уметь читать class of service.

### Средний уровень

- настраивать explicit policies;
- управлять блокерами и aging work;
- балансировать срочные и плановые задачи.

### Продвинутый уровень

- оптимизировать поток через метрики;
- строить прогнозы на основе исторических данных;
- интегрировать Kanban в hybrid governance-контур.

## Основные принципы

- визуализация потока;
- ограничение незавершенной работы (WIP);
- управление потоком по данным;
- непрерывное улучшение.

## Артефакты и роли

| Артефакт | Назначение |
| --- | --- |
| Kanban board | визуализация стадий и задач |
| [WIP](../../glossary.md#abbr-094) limits | ограничение параллельной нагрузки |
| Flow metrics | lead time, cycle time, throughput |
| Service classes | правила приоритета для задач |

Роли обычно легковесные и зависят от команды: service request manager, flow manager, delivery team.

## Классы обслуживания

- Standard;
- Expedite;
- Fixed Date;
- Intangible.

## Шаблон Kanban-элемента

## Плюсы и ограничения

Плюсы:

- высокая гибкость и прозрачный поток;
- быстрое выявление узких мест;
- хорошо подходит для сопровождения и интеграционных команд.

Ограничения:

- сложнее прогнозировать фиксированные даты без статистики;
- без явных политик доска превращается в список задач.

## Пример проекта

Команда интеграций в e-commerce, обрабатывающая смешанный поток задач: партнерские API, инциденты, регрессии.

## Связь с требованиями и рисками

- требования дробятся на минимальные flow-единицы;
- изменения приоритетов отражаются через service classes;
- high-risk задачи ограничиваются WIP-политикой и эскалацией blockers.

## Типичные ошибки

- отсутствие WIP-лимитов;
- игнорирование aging задач;
- нет регулярных flow-review;
