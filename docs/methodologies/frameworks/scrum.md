# Scrum

Scrum - итеративный фреймворк для доставки ценности в фиксированном ритме спринтов.

## Уровни сложности

### Базовый уровень

- знать роли, события и артефакты Scrum;
- понимать цель Sprint Goal;
- уметь читать Product/Sprint backlog.

### Средний уровень

- проводить refinement и planning на основе ценности;
- применять DoR/DoD;
- управлять рисками в рамках спринта.

### Продвинутый уровень

- масштабировать Scrum на несколько команд;
- синхронизировать Scrum с архитектурным governance;
- управлять процессом по метрикам потока и качества.

## Принципы Scrum

- прозрачность;
- инспекция;
- адаптация;
- фокус на ценности.

## Роли и артефакты

| Роль | Ответственность |
| --- | --- |
| Product Owner | ценность и приоритеты |
| Scrum Master | процесс и улучшения |
| Developers | реализация инкремента |

| Артефакт | Назначение |
| --- | --- |
| Product Backlog | полный перечень работы по продукту |
| Sprint Backlog | объем работ спринта |
| Increment | потенциально релизуемый результат |
| Definition of Done | критерии готовности |

## Плюсы и ограничения

Плюсы:

- предсказуемый delivery-ритм;
- быстрый feedback от бизнеса;
- прозрачная приоритизация.

Ограничения:

- плохо работает для хаотичного потока срочных задач;
- требует дисциплины в refinement и DoD;
- без инженерных практик растет техдолг.

## Пример проекта

Мобильный банк с двухнедельным релизным циклом, где приоритеты обновляются на основе продуктовой аналитики.

## Связь с требованиями и рисками

- требования оформляются как backlog items с AC;
- изменения управляются через backlog refinement;
- риски фиксируются и пересматриваются на Planning/Review.

## Типичные ошибки

- спринт без четкого Sprint Goal;
- stories без acceptance criteria;
- ретроспективы без action items;
- перегруз спринта сверх capacity.

## Чек-лист самопроверки

- у спринта есть измеримая цель;
- backlog приоритизирован по бизнес-ценности;
- DoR/DoD реально применяются;
- улучшения из ретро внедряются в следующем спринте.
