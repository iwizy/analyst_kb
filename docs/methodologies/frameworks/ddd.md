# DDD как методологический подход

DDD в методологическом контуре организует работу команд вокруг предметных областей и bounded context.

## Принципы

- ubiquitous language;
- bounded context и context map;
- совместная проработка домена;
- эволюция модели через обратную связь.

## Этапы внедрения

1. Domain discovery (workshops/event storming).
1. Выделение bounded context.
1. Моделирование агрегатов и инвариантов.
1. Настройка интеграций между контекстами.
1. Регулярная ревизия context map.

## Артефакты

- glossary;
- context map;
- domain events catalog;
- model decision log.

## Роли

- domain expert;
- analyst;
- architect;
- product owner;
- engineering team.

## Плюсы и минусы

Плюсы:

- снижает доменные ошибки;
- повышает качество требований;
- упрощает масштабирование по командам.

Минусы:

- требует постоянного участия бизнеса;
- сложен без архитектурной дисциплины.

## Пример подходящего проекта

Fintech-платформа с несколькими доменами: платежи, антифрод, биллинг, отчетность.

## Связь с требованиями и рисками

- требования структурируются по bounded context;
- риски фокусируются на межконтекстной интеграции;
- изменения оцениваются через влияние на доменные инварианты.
