# Lean

Lean в разработке фокусируется на создании ценности и устранении потерь в потоке поставки.

## Уровни сложности

### Базовый уровень

- понимать 5 принципов Lean;
- уметь находить потери в процессе;
- использовать базовые flow-метрики.

### Средний уровень

- строить value stream map;
- формировать backlog улучшений;
- проводить регулярные Kaizen-циклы.

### Продвинутый уровень

- связывать Lean с DevOps и Platform Engineering;
- управлять value stream на уровне нескольких команд;
- измерять экономику потока и эффект изменений.

## Принципы Lean

1. Определить ценность.
2. Картировать поток создания ценности.
3. Обеспечить непрерывный поток.
4. Внедрить pull-механику.
5. Непрерывно совершенствовать систему.

## Артефакты и роли

| Артефакт | Назначение |
| --- | --- |
| Value Stream Map | визуализация потока ценности |
| Waste Register | перечень потерь |
| Improvement Backlog | задачи улучшений |
| Flow Metrics Dashboard | lead time, throughput, flow efficiency |

Роли: product/process owner, team, analyst, Lean facilitator.

## Шаблон Value Stream Map (минимум)

| Шаг потока | Время выполнения | Время ожидания | Узкое место |
| --- | --- | --- | --- |
| Review требований | 1 день | 3 дня | ожидание согласования |

## Плюсы и ограничения

Плюсы:

- сокращение lead time;
- рост предсказуемости потока;
- повышение эффективности всей системы.

Ограничения:

- требуется культура измерения и изменений;
- локальная оптимизация может ухудшать end-to-end поток.

## Пример проекта

Оптимизация релизного процесса корпоративной платформы: уменьшение времени от merge до production с 10 до 3 дней.

## Связь с требованиями и рисками

- приоритизация требований учитывает стоимость задержки;
- риски выявляются в value stream review;
- [NFR](../../glossary.md#abbr-053) включаются в flow policies.

## Типичные ошибки

- улучшения «по ощущениям», без baseline-метрик;
- фокус на локальном [KPI](../../glossary.md#abbr-047) вместо end-to-end результата;
