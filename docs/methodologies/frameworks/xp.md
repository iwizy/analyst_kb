# XP (Extreme Programming)

XP - методика, усиливающая инженерные практики для быстрой и безопасной поставки изменений.

## Уровни сложности

### Базовый уровень

- понимать принципы XP;
- знать практики TDD, pair programming, CI;
- отличать acceptance tests и unit tests.

### Средний уровень

- встраивать XP-практики в Scrum/Kanban;
- проектировать quality gates на уровне кода;
- поддерживать устойчивый темп поставки.

### Продвинутый уровень

- применять XP для high-change доменов;
- управлять техническим долгом через refactoring budget;
- комбинировать XP с DevOps/CI/CD и contract testing.

## Принципы XP

- communication;
- simplicity;
- feedback;
- courage;
- respect.

## Процесс XP (упрощенно)

1. User story + acceptance tests.
2. TDD (Red -> Green -> Refactor).
3. Pair programming / collective ownership.
4. Continuous integration.
5. Small releases.

## Артефакты и роли

| Артефакт | Назначение |
| --- | --- |
| User Story | функциональная цель |
| Acceptance Tests | критерии приемки на уровне поведения |
| Unit Tests | гарантия корректности кода |
| Coding Standards | единый стиль и поддерживаемость |

Роли: заказчик, разработчики, QA/tester, coach.

## Шаблон XP user story

| Поле | Пример |
| --- | --- |
| Story | Как клиент хочу получать код подтверждения за < 15 сек |
| Acceptance Test | сценарий OTP при пиковом трафике |
| Unit Test scope | генерация и валидация OTP |

## Шаблон unit test (минимум)

| Поле | Пример |
| --- | --- |
| Test ID | UT-OTP-001 |
| Method | `validateOtp(code, expireAt)` |
| Given | код не истек и совпадает |
| Expected | `true` |
| Edge case | истекший код -> `false` |

## Плюсы и ограничения

Плюсы:

- низкий дефектный поток;
- высокая скорость обратной связи;
- устойчивость к изменениям требований.

Ограничения:

- высокие требования к инженерной зрелости;
- сложно внедряется фрагментарно;
- требует дисциплины и времени на refactoring.

## Пример проекта

Платежный сервис с частыми изменениями антифрод-правил и высокой ценой регрессии.

## Связь с требованиями и рисками

- требования формулируются через testable examples;
- риски качества снижаются ранней автоматизацией;
- NFR включаются в CI quality gates.

## Типичные ошибки

- TDD «на бумаге» без реального цикла;
- pair programming только для сложных задач, а не как системная практика;
- быстрые релизы без эксплуатационной валидации.

## Чек-лист самопроверки

- unit и acceptance tests покрывают критичные сценарии;
- refactoring выполняется регулярно;
- CI обеспечивает быстрый и надежный feedback;
- история дефектов уменьшается по тренду.
