# V-модель

V-модель усиливает предсказуемый цикл, явно связывая этапы спецификации с соответствующими уровнями тестирования.

## Уровни сложности

### Базовый уровень

- понимать зеркальную связь "спецификация -> тест";
- знать уровни тестирования в V-модели;
- использовать RTM для трассировки.

### Средний уровень

- формировать тест-дизайн параллельно с требованиями;
- управлять quality gates на каждом уровне;
- проводить formal review артефактов.

### Продвинутый уровень

- применять V-модель в safety/compliance средах;
- использовать автоматизацию трассировки и coverage;
- проводить аудит соответствия на уровне программы.

## Этапы и взаимосвязь

| Левое плечо (определение) | Правое плечо (проверка) |
| --- | --- |
| Бизнес-требования | Acceptance testing |
| Системные требования | System testing |
| Архитектурный дизайн | Integration testing |
| Детальный дизайн | Unit testing |

## Плюсы и минусы

Плюсы:

- сильная трассировка требований к тестам;
- ранняя подготовка тестов;
- хорошо подходит для audit/compliance.

Минусы:

- низкая гибкость при частых изменениях;
- высокий documentation overhead;
- риск формального выполнения без продуктовой валидации.

## Пример проекта

Сертифицированный медицинский девайс: фиксированные требования, обязательная полная трассировка и подтверждение качества на каждом уровне тестирования.

## Связь с требованиями, рисками и качеством

- baseline требований фиксируется до кодирования;
- change request проходит impact analysis на все уровни тестирования;
- риски качества выявляются рано через матрицу трассировки.

## Типичные ошибки

- неполный RTM;
- несинхронное обновление требований и тестов;
