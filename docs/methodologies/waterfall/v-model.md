# V-образная модель

V-Model - развитие Waterfall, в котором для каждого этапа разработки заранее
определяется соответствующий уровень тестирования.

## Ключевая идея

Verification и Validation планируются параллельно разработке:

- требования -> приемочные тесты;
- системный дизайн -> системные тесты;
- архитектурный дизайн -> интеграционные тесты;
- кодирование -> модульные тесты.

## Схема соответствия этапов

```kroki-plantuml
@startuml
left to right direction
rectangle "Требования" as DEV_REQ
rectangle "Системный дизайн" as DEV_SYS
rectangle "Архитектурный дизайн" as DEV_ARCH
rectangle "Кодирование" as DEV_CODE
rectangle "Модульные тесты" as TEST_UNIT
rectangle "Интеграционные тесты" as TEST_INT
rectangle "Системные тесты" as TEST_SYS
rectangle "Приемочные тесты" as TEST_ACC

DEV_REQ --> DEV_SYS
DEV_SYS --> DEV_ARCH
DEV_ARCH --> DEV_CODE
DEV_CODE --> TEST_UNIT
TEST_UNIT --> TEST_INT
TEST_INT --> TEST_SYS
TEST_SYS --> TEST_ACC

DEV_REQ ..> TEST_ACC
DEV_SYS ..> TEST_SYS
DEV_ARCH ..> TEST_INT
DEV_CODE ..> TEST_UNIT
@enduml
```

## Преимущества

- тестирование закладывается рано;
- хорошая трассировка "требование -> тест";
- ниже риск пропуска критичных проверок.

## Ограничения

- модель чувствительна к частым изменениям требований;
- при слабой автоматизации растет стоимость тестового контура.

## Чек-лист качества

- Для каждого уровня требований определен набор тестов.
- Тест-дизайн начинается до реализации кода.
- Трассировка требований и тестов поддерживается в актуальном состоянии.
