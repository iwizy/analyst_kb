# Incremental-модель

Incremental-подход делит поставку на последовательные инкременты, каждый из которых проходит полный mini-SDLC и приносит бизнес-ценность.

## Уровни сложности

### Базовый уровень

- понимать идею поэтапной поставки;
- различать инкремент и итерацию;
- использовать roadmap по релизным волнам.

### Средний уровень

- декомпозировать scope на независимые инкременты;
- управлять зависимостями между релизными волнами;
- определять критерии готовности каждого инкремента.

### Продвинутый уровень

- комбинировать Incremental с Spiral для рискованных инициатив;
- управлять архитектурной целостностью при поэтапной поставке;
- оптимизировать release cadence по метрикам.

## Артефакты и роли

| Артефакт | Зачем нужен |
| --- | --- |
| Increment Roadmap | определяет последовательность поставки |
| Release Plan | фиксирует scope и сроки волны |
| Readiness Checklist | критерии выпуска инкремента |
| Feedback Log | данные для следующей волны |

## Плюсы и минусы

Плюсы:

- ранняя поставка полезных функций;
- гибче классического Waterfall;
- удобен для поэтапного внедрения в enterprise.

Минусы:

- риск фрагментации архитектуры;
- сложное управление долгими межинкрементными зависимостями;
- возможен рост интеграционного долга.

## Пример проекта

Внедрение CRM в госсекторе: сначала базовый профиль клиента и регистрация заявок, затем интеграция с внешними реестрами, затем аналитическая отчетность.

## Связь с требованиями и рисками

- требования структурируются по релизным волнам;
- changes внутри текущей волны контролируются строже, чем для будущих волн;
- риски по зависимостям и интеграции пересматриваются перед каждым релизом.

## Типичные ошибки

- неправильная декомпозиция: инкременты сильно зависят друг от друга;
- отсутствие общего архитектурного каркаса;
- перенос неготовых задач между волнами без пересмотра рисков.## Источники
