# Глоссарий

В этом глоссарии собраны основные термины и сокращения, используемые в базе знаний.

## Как пользоваться

- Термины сгруппированы по алфавиту.
- Сокращения приведены отдельно и отсортированы по латинскому алфавиту.
- Если для термина в проекте принят более узкий контекст, ориентируйтесь на профильную статью раздела.

## Термины (А-Я)

| Термин | Определение |
| --- | --- |
| <a id="term-001"></a>Абстракция | Упрощенное представление системы с фокусом на значимых свойствах и скрытием деталей реализации. |
| <a id="term-002"></a>Агрегация | Отношение «часть-целое» между сущностями, при котором часть может существовать отдельно от целого. |
| <a id="term-003"></a>Адаптер | Паттерн/компонент, который преобразует один интерфейс в другой для совместимости систем. |
| <a id="term-080"></a>Активное слушание | Техника коммуникации, при которой собеседника слушают без перебивания, уточняют и перефразируют для точного понимания. |
| <a id="term-004"></a>Алгоритм | Конечная последовательность шагов для решения задачи. |
| <a id="term-005"></a>Альтернатива решения | Один из возможных вариантов реализации цели с собственными ограничениями и рисками. |
| <a id="term-006"></a>Архитектурное решение | Формально зафиксированный выбор по устройству системы и ее эволюции. |
| <a id="term-007"></a>Архитектурный стиль | Типовая организация компонентов и взаимодействий системы (монолит, микросервисы и т.д.). |

| <a id="term-012"></a>Бэклог | Приоритизированный список работ/инициатив для реализации. |
| <a id="term-013"></a>Валидация | Проверка, что решается правильная бизнес-задача и результат нужен пользователю/бизнесу. |
| <a id="term-014"></a>Верификация | Проверка, что артефакт корректно и полно описан по установленным правилам и шаблонам. |
| <a id="term-015"></a>Версионирование | Управление изменениями артефакта через версии с возможностью отслеживания истории. |
| <a id="term-016"></a>Витрина данных | Специализированный слой аналитических данных под конкретные управленческие задачи. |
| <a id="term-017"></a>Воркшоп | Групповая рабочая сессия для совместной проработки и согласования решений. |
| <a id="term-018"></a>Гипотеза | Проверяемое предположение о причине, эффекте или ожидаемом результате изменения. |
| <a id="term-019"></a>Декомпозиция | Разбиение большой задачи/требования на более мелкие управляемые части. |
| <a id="term-082"></a>Дата-мэш (Data Mesh) | Подход к аналитической платформе, где данные управляются как продукт распределенными доменными командами. |
| <a id="term-020"></a>Денормализация | Намеренное дублирование/укрупнение структуры данных для ускорения чтения и аналитики. |
| <a id="term-021"></a>Деплой | Процесс доставки и активации новой версии приложения/компонента в целевой среде. |
| <a id="term-022"></a>Диаграмма | Визуальное представление структуры, процесса или взаимодействий системы. |
| <a id="term-023"></a>Домен | Предметная область, в рамках которой описывается бизнес-контекст и правила. |
| <a id="term-024"></a>Документация как код (Docs as Code) | Подход к ведению документации в репозитории с версионированием и review-процессом. |
| <a id="term-025"></a>Доступность | Доля времени, когда система доступна и выполняет заявленную функцию. |
| <a id="term-026"></a>Жизненный цикл ПО | Набор фаз от инициации и разработки до эксплуатации и вывода из эксплуатации. |
| <a id="term-027"></a>Зависимость | Связь между задачами/компонентами, при которой выполнение одного элемента влияет на другой. |
| <a id="term-028"></a>Задача | Минимальная единица планируемой работы с владельцем и ожидаемым результатом. |
| <a id="term-029"></a>Идемпотентность | Свойство операции давать тот же результат при повторном выполнении с теми же входными данными. |
| <a id="term-030"></a>Инвариант | Условие, которое должно оставаться истинным при любых допустимых состояниях процесса/системы. |
| <a id="term-031"></a>Индекс | Структура данных в СУБД, ускоряющая поиск/фильтрацию по полям. |
| <a id="term-032"></a>Инкремент | Завершенная и потенциально поставляемая часть продукта после итерации. |
| <a id="term-033"></a>Интеграция | Организация взаимодействия между системами, сервисами или компонентами. |
| <a id="term-034"></a>Инцидент | Незапланированное событие, ухудшающее качество сервиса или доступность системы. |
| <a id="term-035"></a>Капасити (Capacity) | Реальная доступная емкость команды/системы для выполнения работы в периоде. |
| <a id="term-036"></a>Кейс | Практическая ситуация или пример применения подхода в конкретном контексте. |
| <a id="term-037"></a>Контекстная диаграмма | Верхнеуровневая схема границ системы и взаимодействия с внешними участниками/системами. |
| <a id="term-038"></a>Контракт | Формально согласованное описание интерфейса взаимодействия (например, API-контракт). |
| <a id="term-083"></a>Контракт-фёрст (Contract-First) | Подход, при котором сначала проектируется и утверждается контракт (например, API), а затем пишется реализация. |
| <a id="term-084"></a>Код-фёрст (Code-First) | Подход, при котором интерфейс и спецификация формируются на основе уже написанного кода. |
| <a id="term-039"></a>Конфигурация | Набор параметров, определяющих поведение системы в среде. |
| <a id="term-040"></a>Консистентность | Согласованность данных между узлами/компонентами системы. |
| <a id="term-041"></a>Критерии приемки | Набор проверяемых условий, при выполнении которых требование считается реализованным. |
| <a id="term-042"></a>Латентность | Время отклика системы или задержка передачи/обработки данных. |
| <a id="term-043"></a>Маршрутизация | Выбор пути обработки запроса/сообщения между компонентами системы. |
| <a id="term-044"></a>Метрика | Измеримый показатель, используемый для оценки состояния, прогресса или качества. |
| <a id="term-045"></a>Миграция | Управляемое изменение структуры данных, платформы или системы с сохранением работоспособности. |
| <a id="term-046"></a>Нагрузка | Интенсивность обращений к системе (запросы, события, объем данных, пользователи). |
| <a id="term-047"></a>Нефункциональное требование | Требование к качественным характеристикам системы (производительность, безопасность и т.п.). |
| <a id="term-048"></a>Нормализация | Проектирование структуры данных для устранения избыточности и аномалий обновления. |
| <a id="term-049"></a>Обратная совместимость | Способность новой версии системы корректно работать со старыми клиентами/контрактами. |
| <a id="term-085"></a>Обсервабилити (Observability) | Набор практик и инструментов, позволяющих объяснять внутреннее состояние системы по внешним сигналам (метрики, логи, трассировки). |
| <a id="term-050"></a>Ограничение | Условие, сужающее пространство решений (техническое, бизнесовое, правовое и т.д.). |
| <a id="term-051"></a>Оркестрация | Централизованное управление шагами процесса с явной координацией операций. |
| <a id="term-052"></a>Паттерн | Повторно используемое типовое решение для класса задач. |
| <a id="term-053"></a>Производительность | Способность системы обрабатывать нагрузку за заданное время и ресурс. |
| <a id="term-054"></a>Прототип | Быстрая модель решения для проверки гипотез и сценариев до реализации. |
| <a id="term-055"></a>Регрессия | Ухудшение ранее корректного поведения после изменения системы. |

| <a id="term-057"></a>Ретроспектива | Регулярный разбор прошедшего периода для выявления улучшений процесса. |
| <a id="term-058"></a>Роадмап | План развития продукта/системы по инициативам и временным горизонтам. |
| <a id="term-059"></a>Сага | Паттерн управления распределенной бизнес-транзакцией через шаги и компенсации. |
| <a id="term-060"></a>Сервис | Самостоятельный компонент, предоставляющий функциональность через интерфейс. |
| <a id="term-061"></a>Сервисная шина | Интеграционный слой для маршрутизации и преобразования сообщений между системами. |
| <a id="term-062"></a>Синхронное взаимодействие | Модель, при которой вызывающая сторона ожидает ответ в рамках текущего запроса. |

| <a id="term-064"></a>Событие | Факт изменения состояния, который может запускать реакцию в системе. |
| <a id="term-065"></a>Спецификация | Формализованное описание требований, интерфейсов или поведения системы. |
| <a id="term-066"></a>Стейкхолдер | Заинтересованная сторона, влияющая на решение или затрагиваемая его результатом. |
| <a id="term-067"></a>Схема данных | Формальная структура хранения и связей данных. |
| <a id="term-068"></a>Технический долг | Накопленные упрощения/компромиссы, увеличивающие стоимость дальнейших изменений. |
| <a id="term-069"></a>Трассировка | Связь требований с реализацией, тестами и артефактами для контроля покрытия и влияния изменений. |
| <a id="term-070"></a>Транзакция | Последовательность операций, выполняемая как единое целое по правилам целостности. |
| <a id="term-086"></a>Томас-Килман (Thomas-Kilmann Model) | Модель стилей поведения в конфликте: конкуренция, сотрудничество, компромисс, избегание, приспособление. |
| <a id="term-087"></a>Три Амигос (Three Amigos) | Формат совместной проработки требований и сценариев с участием аналитика/PO, разработки и QA. |

| <a id="term-072"></a>Триггер | Условие/событие, запускающее действие или переход состояния. |
