# Глоссарий

В этом глоссарии собраны основные термины и сокращения, используемые в базе знаний.

## Как пользоваться

- Термины сгруппированы по алфавиту.
- Сокращения приведены отдельно и отсортированы по латинскому алфавиту.
- Если для термина в проекте принят более узкий контекст, ориентируйтесь на профильную статью раздела.

## Термины (А-Я)

| Термин | Определение |
| --- | --- |
| Абстракция | Упрощенное представление системы с фокусом на значимых свойствах и скрытием деталей реализации. |
| Агрегация | Отношение «часть-целое» между сущностями, при котором часть может существовать отдельно от целого. |
| Адаптер | Паттерн/компонент, который преобразует один интерфейс в другой для совместимости систем. |
| Алгоритм | Конечная последовательность шагов для решения задачи. |
| Альтернатива решения | Один из возможных вариантов реализации цели с собственными ограничениями и рисками. |
| Архитектурное решение | Формально зафиксированный выбор по устройству системы и ее эволюции. |
| Архитектурный стиль | Типовая организация компонентов и взаимодействий системы (монолит, микросервисы и т.д.). |
| Базовая линия (Baseline) | Согласованная и зафиксированная версия артефакта, от которой ведутся изменения. |
| Баг (дефект) | Несоответствие фактического поведения системы ожидаемому. |
| Бизнес-правило | Формализованное правило предметной области, ограничивающее или направляющее поведение системы. |
| Бизнес-требование | Требование, выражающее цель или ограничение бизнеса, которое должно быть поддержано решением. |
| Бэклог | Приоритизированный список работ/инициатив для реализации. |
| Валидация | Проверка, что решается правильная бизнес-задача и результат нужен пользователю/бизнесу. |
| Верификация | Проверка, что артефакт корректно и полно описан по установленным правилам и шаблонам. |
| Версионирование | Управление изменениями артефакта через версии с возможностью отслеживания истории. |
| Витрина данных | Специализированный слой аналитических данных под конкретные управленческие задачи. |
| Воркшоп | Групповая рабочая сессия для совместной проработки и согласования решений. |
| Гипотеза | Проверяемое предположение о причине, эффекте или ожидаемом результате изменения. |
| Декомпозиция | Разбиение большой задачи/требования на более мелкие управляемые части. |
| Денормализация | Намеренное дублирование/укрупнение структуры данных для ускорения чтения и аналитики. |
| Деплой | Процесс доставки и активации новой версии приложения/компонента в целевой среде. |
| Диаграмма | Визуальное представление структуры, процесса или взаимодействий системы. |
| Домен | Предметная область, в рамках которой описывается бизнес-контекст и правила. |
| Документация как код (Docs as Code) | Подход к ведению документации в репозитории с версионированием и review-процессом. |
| Доступность | Доля времени, когда система доступна и выполняет заявленную функцию. |
| Жизненный цикл ПО | Набор фаз от инициации и разработки до эксплуатации и вывода из эксплуатации. |
| Зависимость | Связь между задачами/компонентами, при которой выполнение одного элемента влияет на другой. |
| Задача | Минимальная единица планируемой работы с владельцем и ожидаемым результатом. |
| Идемпотентность | Свойство операции давать тот же результат при повторном выполнении с теми же входными данными. |
| Инвариант | Условие, которое должно оставаться истинным при любых допустимых состояниях процесса/системы. |
| Индекс | Структура данных в СУБД, ускоряющая поиск/фильтрацию по полям. |
| Инкремент | Завершенная и потенциально поставляемая часть продукта после итерации. |
| Интеграция | Организация взаимодействия между системами, сервисами или компонентами. |
| Инцидент | Незапланированное событие, ухудшающее качество сервиса или доступность системы. |
| Капасити (Capacity) | Реальная доступная емкость команды/системы для выполнения работы в периоде. |
| Кейс | Практическая ситуация или пример применения подхода в конкретном контексте. |
| Контекстная диаграмма | Верхнеуровневая схема границ системы и взаимодействия с внешними участниками/системами. |
| Контракт | Формально согласованное описание интерфейса взаимодействия (например, API-контракт). |
| Конфигурация | Набор параметров, определяющих поведение системы в среде. |
| Консистентность | Согласованность данных между узлами/компонентами системы. |
| Критерии приемки | Набор проверяемых условий, при выполнении которых требование считается реализованным. |
| Латентность | Время отклика системы или задержка передачи/обработки данных. |
| Маршрутизация | Выбор пути обработки запроса/сообщения между компонентами системы. |
| Метрика | Измеримый показатель, используемый для оценки состояния, прогресса или качества. |
| Миграция | Управляемое изменение структуры данных, платформы или системы с сохранением работоспособности. |
| Нагрузка | Интенсивность обращений к системе (запросы, события, объем данных, пользователи). |
| Нефункциональное требование | Требование к качественным характеристикам системы (производительность, безопасность и т.п.). |
| Нормализация | Проектирование структуры данных для устранения избыточности и аномалий обновления. |
| Обратная совместимость | Способность новой версии системы корректно работать со старыми клиентами/контрактами. |
| Ограничение | Условие, сужающее пространство решений (техническое, бизнесовое, правовое и т.д.). |
| Оркестрация | Централизованное управление шагами процесса с явной координацией операций. |
| Паттерн | Повторно используемое типовое решение для класса задач. |
| Производительность | Способность системы обрабатывать нагрузку за заданное время и ресурс. |
| Прототип | Быстрая модель решения для проверки гипотез и сценариев до реализации. |
| Регрессия | Ухудшение ранее корректного поведения после изменения системы. |
| Репликация | Копирование данных между узлами для отказоустойчивости, масштабирования или аналитики. |
| Ретроспектива | Регулярный разбор прошедшего периода для выявления улучшений процесса. |
| Роадмап | План развития продукта/системы по инициативам и временным горизонтам. |
| Сага | Паттерн управления распределенной бизнес-транзакцией через шаги и компенсации. |
| Сервис | Самостоятельный компонент, предоставляющий функциональность через интерфейс. |
| Сервисная шина | Интеграционный слой для маршрутизации и преобразования сообщений между системами. |
| Синхронное взаимодействие | Модель, при которой вызывающая сторона ожидает ответ в рамках текущего запроса. |
| Скоуп (Scope) | Границы работ и результатов, входящих в текущую поставку/инициативу. |
| Событие | Факт изменения состояния, который может запускать реакцию в системе. |
| Спецификация | Формализованное описание требований, интерфейсов или поведения системы. |
| Стейкхолдер | Заинтересованная сторона, влияющая на решение или затрагиваемая его результатом. |
| Схема данных | Формальная структура хранения и связей данных. |
| Технический долг | Накопленные упрощения/компромиссы, увеличивающие стоимость дальнейших изменений. |
| Трассировка | Связь требований с реализацией, тестами и артефактами для контроля покрытия и влияния изменений. |
| Транзакция | Последовательность операций, выполняемая как единое целое по правилам целостности. |
| Требование | Формализованное ожидание к системе, процессу или продукту. |
| Триггер | Условие/событие, запускающее действие или переход состояния. |
| Устойчивость | Способность системы сохранять работоспособность при сбоях и пиковых нагрузках. |
| Фасилитация | Управление групповой дискуссией для достижения рабочего результата. |
| Фича (Feature) | Пользовательски значимая функциональность продукта. |
| Хореография | Децентрализованная координация, где участники реагируют на события без центрального оркестратора. |
| Целевая архитектура | Желаемое целевое состояние архитектуры на заданном горизонте развития. |
| Шардирование | Горизонтальное разделение данных на независимые части (шарды). |
| Эпик | Крупная единица продуктовой работы, объединяющая связанные user stories/задачи. |

## Сокращения (A-Z)

| Сокращение | Расшифровка | Краткое значение |
| --- | --- | --- |
| A/B | A/B Testing | Эксперимент сравнения двух вариантов решения. |
| AC | Acceptance Criteria | Критерии приемки требования/истории. |
| ADR | Architecture Decision Record | Документ фиксации архитектурного решения. |
| AHP | Analytic Hierarchy Process | Метод приоритизации через парные сравнения. |
| API | Application Programming Interface | Интерфейс программного взаимодействия систем. |
| APM | Application Performance Monitoring | Мониторинг производительности приложений. |
| BDD | Behavior-Driven Development | Подход к разработке через описание поведения. |
| BI | Business Intelligence | Аналитика и визуализация бизнес-данных. |
| BPMN | Business Process Model and Notation | Нотация моделирования бизнес-процессов. |
| BRD | Business Requirements Document | Документ бизнес-требований. |
| C4 | Context, Container, Component, Code | Модель описания архитектуры на четырех уровнях. |
| CAC | Customer Acquisition Cost | Стоимость привлечения клиента. |
| CAP | Consistency, Availability, Partition Tolerance | Теорема компромиссов распределенных систем. |
| CDC | Change Data Capture | Захват изменений данных из источников. |
| CD3 | Cost of Delay Divided by Duration | Метод приоритизации по задержке и длительности. |
| CI | Continuous Integration | Непрерывная интеграция изменений в кодовую базу. |
| CI/CD | Continuous Integration / Continuous Delivery (Deployment) | Сквозной процесс сборки, тестирования и доставки. |
| CMMI | Capability Maturity Model Integration | Модель зрелости процессов разработки. |
| COBIT | Control Objectives for Information and Related Technologies | Фреймворк управления ИТ и ИТ-контролями. |
| CQRS | Command Query Responsibility Segregation | Разделение операций записи и чтения. |
| CRUD | Create, Read, Update, Delete | Базовые операции работы с данными. |
| CSV | Comma-Separated Values | Текстовый табличный формат данных. |
| DACI | Driver, Approver, Contributors, Informed | Модель распределения ролей в принятии решения. |
| DDD | Domain-Driven Design | Подход проектирования через доменную модель. |
| DFD | Data Flow Diagram | Диаграмма потоков данных. |
| DoD | Definition of Done | Критерии завершенности работы. |
| DoR | Definition of Ready | Критерии готовности элемента к началу работ. |
| DS | Data Science | Дисциплина анализа данных и построения моделей. |
| DWH | Data Warehouse | Хранилище аналитических данных. |
| ELT | Extract, Load, Transform | Подход загрузки с последующей трансформацией. |
| EPC / eEPC | Event-driven Process Chain | Нотация моделирования процессов на событиях и функциях. |
| ERD | Entity Relationship Diagram | Диаграмма сущностей и связей данных. |
| ETL | Extract, Transform, Load | Подход извлечения, преобразования и загрузки данных. |
| FMEA | Failure Mode and Effects Analysis | Метод анализа отказов и их последствий. |
| FTR | First Time Right | Показатель выполнения без повторной доработки. |
| gRPC | Google Remote Procedure Call | Протокол высокопроизводительных RPC-вызовов. |

| IDE | Integrated Development Environment | Интегрированная среда разработки. |

| IRR | Internal Rate of Return | Внутренняя норма доходности проекта. |
| ISO | International Organization for Standardization | Международная организация по стандартизации. |
| ITIL | Information Technology Infrastructure Library | Набор практик управления ИТ-сервисами. |

| JWT | JSON Web Token | Формат подписанного токена для аутентификации/авторизации. |

| LTV | Lifetime Value | Доходность клиента за весь срок взаимодействия. |

| mTLS | mutual TLS | Двусторонняя TLS-аутентификация сторон. |
| NFR | Non-Functional Requirement | Нефункциональное требование. |
| NPS | Net Promoter Score | Индекс готовности рекомендовать продукт/сервис. |
| NPV | Net Present Value | Чистая приведенная стоимость проекта. |
| OAuth | Open Authorization | Протокол авторизации доступа к ресурсам. |
| OIDC | OpenID Connect | Надстройка над OAuth для аутентификации. |
| OKR | Objectives and Key Results | Система постановки целей и измеримых результатов. |
| OLAP | Online Analytical Processing | Аналитическая обработка данных. |
| OLTP | Online Transaction Processing | Оперативная транзакционная обработка. |
| PACELC | Partition, Availability, Consistency, Else Latency/Consistency | Расширение CAP с учетом задержек в нормальном режиме. |
| PERT | Program Evaluation and Review Technique | Метод оценки сроков через оптимистичный/вероятный/пессимистичный сценарии. |
| PII | Personally Identifiable Information | Персональные данные, позволяющие идентифицировать человека. |
| PMBOK | Project Management Body of Knowledge | Свод знаний по управлению проектами. |
| PO | Product Owner | Роль, отвечающая за ценность и приоритизацию продукта. |
| PRD | Product Requirements Document | Документ продуктовых требований. |
| QA | Quality Assurance | Практики и роль обеспечения качества. |
| RACI | Responsible, Accountable, Consulted, Informed | Матрица распределения ответственности. |
| RBAC | Role-Based Access Control | Модель управления доступом на основе ролей. |
| RCA | Root Cause Analysis | Анализ первопричины проблемы/инцидента. |

| RPS | Requests Per Second | Количество запросов в секунду. |
| RTO | Recovery Time Objective | Целевое время восстановления сервиса. |
| SAFe | Scaled Agile Framework | Фреймворк масштабирования Agile-практик. |
| SAGA | Saga Pattern | Паттерн распределенных транзакций с компенсациями. |
