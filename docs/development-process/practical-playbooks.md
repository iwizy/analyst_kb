# Практические playbook’и процесса разработки
Страница дает прикладные сценарии настройки delivery-процесса под разные типы проектов: fintech, e-commerce, гос‑контур и внутренние платформенные продукты.

## Уровни сложности

### Базовый уровень

- выбрать базовый процесс: Scrum, Kanban или гибрид;
- определить минимальные артефакты и ритмы команды;
- согласовать Definition of Ready/Done.

### Средний уровень

- настроить quality gates между этапами;
- связать требования, тестирование, релизы и эксплуатацию;
- выстроить регулярный цикл управления рисками.

### Продвинутый уровень

- управлять процессом нескольких команд;
- внедрить метрики потока и надежности (lead time, deployment frequency, [MTTR](../glossary.md#abbr-051));
- адаптировать процесс под регуляторику и аудит.

## Playbook 1. Fintech-продукт

### Цель

Обеспечить быстрый выпуск изменений без нарушения требований комплаенса и безопасности.

### Рекомендуемый контур

| Блок | Практика |
| --- | --- |
| Управление требованиями | формальные критерии согласования + трассировка |
| Разработка | Scrum с обязательным refinement и архитектурным review |
| Качество | автоматические тесты + security scanning в [CI/CD](../glossary.md#abbr-017) |
| Релизы | canary/blue-green для критичных сервисов |
| Эксплуатация | 24/7 мониторинг, postmortem на инциденты |

## Playbook 2. E-commerce с быстрыми экспериментами

### Цель

Ускорить поставку фич и A/B-экспериментов без деградации стабильности.

### Рекомендуемый контур

| Блок | Практика |
| --- | --- |
| Процесс | Kanban или Scrum+Kanban |
| Приоритизация | бизнес-метрики + краткие discovery-сессии |
| Тестирование | shift-left + автотесты на критические сценарии |
| Релизы | frequent release + feature flags |
| Метрики | conversion, lead time, error rate, rollback rate |

## Playbook 3. Гос‑проект с регламентным контролем

### Цель

Соблюсти формальные согласования и сроки поставки при большом числе стейкхолдеров.

### Рекомендуемый контур

| Блок | Практика |
| --- | --- |
| Модель | Water-Scrum-Fall |
| Документация | формализованные спецификации и протоколы решений |
| Контроль качества | stage-gate приемка по заранее утвержденным критериям |
| Управление изменениями | централизованный workflow Change Request |
| Эксплуатация | регламентные runbook и контроль [SLA](../glossary.md#abbr-079) |

## Playbook 4. Внутренняя платформенная команда

### Цель

Поддерживать стабильную платформу и параллельно развивать внутренние сервисы.

### Рекомендуемый контур

- Kanban с классами обслуживания;
- отдельная очередь incident/problem/change;
- SRE-практики для ключевых сервисов;
- ежемесячный процессный review с метриками качества.

## Шаблон внедрения процесса (90 дней)

| Период | Фокус |
| --- | --- |
| 1-30 день | диагностика текущего процесса и выбор модели |
| 31-60 день | запуск ритмов, шаблонов, quality gates |
| 61-90 день | стабилизация метрик, корректировка bottleneck’ов |

## Минимальные артефакты для старта

- backlog и правила приоритизации;
- DoR/DoD;
- release checklist и rollback-план;
- risk register;
- decision log по процессным изменениям.

## Типичные ошибки

- запуск сложного процесса без пилота;
- отсутствие явных владельцев ритуалов и артефактов;
- перегрузка команды документами без понятной ценности;
- метрики без регулярных действий по результатам.
