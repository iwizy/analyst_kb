# Разбор инцидентов

Разбор инцидентов (post-incident analysis) помогает устранить первопричину
сбоя и снизить вероятность повторения.

## Цель разбора

- понять, почему произошел инцидент;
- оценить влияние на пользователей и бизнес;
- сформировать корректирующие и профилактические меры.

## Типовой процесс

```kroki-plantuml
@startuml
start
:Зафиксировать факты инцидента;
:Собрать таймлайн событий;
:Определить первопричину;
:Сформировать corrective actions;
:Назначить ответственных и сроки;
:Проверить выполнение мер;
stop
@enduml
```

## Что включать в postmortem

- краткое описание инцидента и impact;
- хронологию обнаружения и восстановления;
- первопричину и факторы усиления;
- план действий с владельцами и датами.

## Типичные ошибки

- поиск виноватого вместо анализа системы;
- отсутствие измеримых follow-up действий;
- закрытие инцидента без проверки эффекта мер.

## Чек-лист качества

- Разбор выполнен по фактам и подтвержденным данным.
- Для действий указаны владельцы, сроки и критерии завершения.
- Результаты постмортема отражены в backlog и стандартах работы.

## Практическое дополнение

- Проведите мини-аудит текущего процесса по схеме: `требование -> реализация -> тест -> эксплуатация`.
- Зафиксируйте одно узкое место и добавьте улучшение в ближайший спринт/итерацию.
- Проверьте эффект через 2 недели по измеримой метрике (дефекты, lead time, MTTR или SLA).
