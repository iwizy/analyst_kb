# Тест-кейсы

Тест-кейс это формализованный сценарий проверки, который подтверждает соответствие решения требованиям и критериям приемки.

## Уровни сложности

### Базовый уровень

- составлять понятный и воспроизводимый тест-кейс;
- разделять предусловия, шаги и ожидаемые результаты;
- поддерживать связь тест-кейса с требованием.

### Средний уровень

- проектировать позитивные, негативные и граничные сценарии;
- группировать тесты по рискам и приоритетам;
- подготавливать тестовые наборы данных.

### Продвинутый уровень

- использовать шаблоны [BDD](../../glossary.md#abbr-007) и test design techniques;
- оценивать полноту покрытия по RTM;
- автоматизировать повторяемые сценарии.

## Шаблон тест-кейса

| Поле | Описание |
| --- | --- |
| ID | уникальный идентификатор |
| Requirement ID | ссылка на требование |
| Цель | что проверяем |
| Предусловия | что должно быть до старта |
| Шаги | последовательность действий |
| Ожидаемый результат | измеримый outcome |
| Приоритет | Critical/High/Medium/Low |
| Статус | Passed/Failed/Blocked |

## Пример заполнения

| Поле | Значение |
| --- | --- |
| ID | TC-ORD-[AC](../../glossary.md#abbr-002)-001 |
| Requirement ID | US-ORDER-012 |
| Цель | Создание заказа зарегистрированным пользователем |
| Предусловия | Пользователь авторизован, товар в наличии |
| Шаги | 1) Добавить товар в корзину 2) Нажать "Оформить" |
| Ожидаемый результат | Заказ создан, статус `NEW`, ответ [API](../../glossary.md#abbr-005) 201 |
| Приоритет | High |
| Статус | Passed |

## Given/When/Then формат

```gherkin
Given пользователь авторизован и товар в наличии
When пользователь оформляет заказ
Then система создает заказ со статусом NEW
```

## Частые ошибки

- несколько независимых проверок в одном кейсе;
- размытый expected result ("все работает корректно");
- отсутствие связи с требованиями;
- неактуальные тест-кейсы после изменения функциональности.

## Практические рекомендации

- один тест-кейс = один главный проверяемый результат;
- отдельно вести happy-path и error-path;
- обновлять тест-кейсы в том же цикле, что и требования;
- использовать единые naming conventions.
