# Приемка функциональности ([UAT](../../glossary.md#abbr-089))

Приемка подтверждает, что инкремент соответствует бизнес-ожиданиям, требованиям и ограничениям среды эксплуатации.

## Уровни сложности

### Базовый уровень

- понимать цель UAT и отличие от system testing;
- проверять сценарии по согласованным [AC](../../glossary.md#abbr-002);
- фиксировать формальное решение о приемке.

### Средний уровень

- проводить UAT-сессии с бизнесом и пользователями;
- управлять замечаниями и условной приемкой;
- учитывать [NFR](../../glossary.md#abbr-053) и операционные критерии готовности.

### Продвинутый уровень

- выстраивать risk-based приемку для крупных релизов;
- включать эксплуатационные команды в go/no-go;
- использовать метрики приемки для улучшения требований.

## Процесс UAT

```kroki-plantuml
skinparam monochrome true
start
:Согласовать AC и scope приемки;
:Подготовить окружение и данные;
:Провести UAT-сценарии;
if (Критерии выполнены?) then (Да)
  :Подписать протокол приемки;
  :Go для релиза;
else (Нет)
  :Сформировать замечания;
  :Повторная проверка;
endif
stop
```

## Участники [UAT](../../glossary.md#abbr-089)

| Роль | Ответственность |
| --- | --- |
| Бизнес-владелец/[PO](../../glossary.md#abbr-065) | подтверждение бизнес-ценности |
| BA/SA | трассировка [AC](../../glossary.md#abbr-002) и протоколирование решений |
| [QA](../../glossary.md#abbr-067) | организация сценариев и evidence |
| Dev/DevOps | оперативные исправления и техническая поддержка |

## Шаблон протокола приемки

| Поле | Содержание |
| --- | --- |
| Release ID | версия и дата |
| Scope | список user stories/use cases |
| AC статус | выполнено/частично/не выполнено |
| Замечания | список open issues |
| Решение | accepted / conditionally accepted / rejected |
| Подписи | бизнес, QA, BA/SA, техлид |

## Типовые ошибки

- приемка начинается без согласованных AC;
- подмена приемки демонстрацией без проверок;
- отсутствие формального решения и условий доработки;
