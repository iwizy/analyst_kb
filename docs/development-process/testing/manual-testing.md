# Ручное тестирование

Ручное тестирование особенно важно там, где нужна экспертная оценка поведения системы: UX, сложные бизнес-процессы, интеграционные отклонения, нестандартные пользовательские сценарии.

## Уровни сложности

### Базовый уровень

- выполнять тест-кейсы и фиксировать результаты;
- корректно оформлять дефекты;
- проверять ключевые пользовательские сценарии.

### Средний уровень

- проводить exploratory testing;
- проверять интеграционные и регрессионные цепочки;
- формировать рекомендации по автоматизации сценариев.

### Продвинутый уровень

- планировать risk-based manual testing;
- комбинировать ручные и автоматизированные проверки;
- выстраивать процесс приемки с бизнесом и эксплуатацией.

## Когда ручное тестирование особенно полезно

- ранние прототипы и новые UX-потоки;
- сложные интеграции с нестабильными внешними сервисами;
- сценарии, где важны визуальные и текстовые аспекты;
- проверка recovery/fallback поведения.

## Типовой процесс

1. Подготовка окружения и данных.
2. Прогон тест-кейсов и exploratory-чек-листа.
3. Фиксация дефектов (severity, priority, шаги, evidence).
4. Ретест и regression check.

## Шаблон дефект-репорта

| Поле | Пример |
| --- | --- |
| Defect ID | BUG-2026-144 |
| Связь с требованием | US-RET-022 |
| Шаги воспроизведения | 1..N |
| Ожидаемый результат | сумма возврата рассчитывается автоматически |
| Фактический результат | сумма не проставляется |
| Severity/Priority | High/High |
| Окружение | staging v2.4.1 |

## Типовые ошибки

- проверка только happy path;
- слабое описание дефекта;
- отсутствие критериев завершения ручного тестирования;
