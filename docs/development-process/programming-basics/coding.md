# Написание кода

Даже базовое понимание практик написания кода помогает аналитику формулировать
реализуемые требования и повышать качество коммуникации с командой.

## Базовые практики

- small changes: небольшие логические изменения;
- читаемость: понятные имена и структура;
- явные контракты: валидация входов, предсказуемые выходы;
- тестируемость: код можно проверить автоматически.

## Что учитывать в требованиях

- предусматривать edge cases и ошибки;
- явно фиксировать инварианты и правила;
- избегать двусмысленностей в терминах и состояниях.

## Пример жизненного цикла изменения

```kroki-plantuml
@startuml
start
:Понять требование;
:Подготовить изменение кода;
:Добавить/обновить тесты;
:Открыть PR;
if (Ревью пройдено?) then (Да)
  :Merge;
  :Проверить в целевой среде;
else (Нет)
  :Исправить замечания;
endif
stop
@enduml
```

## Типичные ошибки

- реализация без автоматических проверок;
- большие PR с разнородными изменениями;
- отсутствие описания технических ограничений в задаче.

## Чек-лист качества

- Изменение покрывает требуемый сценарий и ошибки.
- Код-ревью проходит по понятным критериям.
- Технический долг, созданный изменением, зафиксирован явно.

## Практическое дополнение

- Проведите мини-аудит текущего процесса по схеме: `требование -> реализация -> тест -> эксплуатация`.
- Зафиксируйте одно узкое место и добавьте улучшение в ближайший спринт/итерацию.
- Проверьте эффект через 2 недели по измеримой метрике (дефекты, lead time, MTTR или SLA).
