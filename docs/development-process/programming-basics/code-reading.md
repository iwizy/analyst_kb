# Чтение кода: практический подход для аналитика

Чтение кода помогает проверять соответствие реализации требованиям, выявлять пробелы в AC и быстрее диагностировать дефекты.

## Уровни сложности

### Базовый уровень

- находить entry-point пользовательского сценария;
- отличать бизнес-логику от инфраструктурного кода;
- проверять happy path и ключевые error-path.

### Средний уровень

- читать цепочку вызовов между сервисами;
- анализировать условия, валидации и обработку ошибок;
- сопоставлять требования с тестами и логами.

### Продвинутый уровень

- находить скрытые риски: race conditions, idempotency gaps, data inconsistency;
- анализировать влияние изменений между модулями;
- формировать технически точные change requests.

## Алгоритм чтения кода

1. Взять конкретный пользовательский сценарий.
2. Найти входную точку (endpoint/handler/controller).
3. Пройти путь данных до сохранения/ответа.
4. Проверить негативные ветки и исключения.
5. Сопоставить поведение с AC и NFR.

## Что анализировать в первую очередь

| Область | Вопрос |
| --- | --- |
| Валидация входа | какие проверки блокируют некорректные данные? |
| Бизнес-правила | где реализованы ключевые условия предметной области? |
| Интеграции | что произойдет при таймауте или 5xx от внешнего API? |
| Логирование | хватит ли данных для RCA? |

## Мини-пример проверки условия

```java
if (order.totalAmount().compareTo(MAX_LIMIT) > 0) {
    throw new BusinessRuleException("LIMIT_EXCEEDED");
}
```

Что проверить аналитику:

- лимит согласован с бизнес-правилом;
- код ошибки отражен в API-контракте;
- ошибка покрыта тест-кейсом и сценариями приемки.

## Типовые ошибки при чтении кода

- проверка только happy path;
- игнорирование feature flags и конфигурации;
- выводы без сопоставления с runtime-логами;
- анализ без учета версий API и схем данных.## Источники

- Clean Code (R. Martin).
- Refactoring (M. Fowler).
- Effective Java (J. Bloch).
