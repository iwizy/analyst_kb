# Документирование API

Документирование API фиксирует контракт интеграции: что вызывается, в каком формате и какой результат ожидается. Без качественной документации интеграции становятся дорогими в поддержке и часто ломаются при изменениях.

## Что обязательно должно быть в документации

- назначение API и область применения;
- схема авторизации;
- базовый URL и версии;
- перечень эндпоинтов с методами;
- параметры запроса и формат тела;
- формат ответа и коды ошибок;
- ограничения (rate limits, таймауты, квоты);
- примеры запросов и ответов.

## Рекомендуемая структура описания эндпоинта

1. Цель эндпоинта.
2. HTTP-метод и URL.
3. Требования к авторизации.
4. Параметры запроса.
5. Пример запроса.
6. Пример успешного ответа.
7. Список ошибок и условия их возникновения.

## Пример

### Получение списка задач

- Метод: `GET`
- URL: `/tasks`
- Авторизация: `Bearer Token`
- Параметры: `status`, `limit`

```http
GET /v1/tasks?status=completed&limit=10
Authorization: Bearer <token>
```

```json
{
  "tasks": [
    {"id": 1, "title": "Купить продукты", "status": "completed"},
    {"id": 2, "title": "Закончить проект", "status": "in_progress"}
  ]
}
```

## Инструменты

- Swagger/OpenAPI для контрактов и интерактивной документации.
- Postman для коллекций и тестирования запросов.
- ReDoc для публикации читаемой документации по OpenAPI.

## Типичные ошибки

- документация отстает от фактической реализации;
- нет примеров ошибок;
- не описаны ограничения и лимиты;
- отсутствуют правила версионирования.

## Итог

Качественная API-документация снижает стоимость интеграций и ускоряет подключение новых потребителей.
