# Основные уязвимости веб-сервисов и приложений

> Глоссарий терминов и сокращений: [Открыть](../../glossary.md)


Большинство уязвимостей повторяются из проекта в проект. Для аналитика важно
учитывать их на этапе требований и архитектурных решений, а не только на
пентесте перед релизом.

## Частые классы уязвимостей

| Уязвимость | Пример риска | Базовая защита |
| --- | --- | --- |
| Injection (SQL/NoSQL/Command) | выполнение чужого кода/запросов | параметризованные запросы, валидация ввода |
| Broken Access Control | доступ к чужим данным | server-side проверки прав на каждую операцию |
| XSS | внедрение скрипта в браузер | экранирование, CSP, безопасный рендеринг |
| CSRF | выполнение действий от лица пользователя | CSRF-токены, SameSite cookies |
| Security Misconfiguration | компрометация из-за дефолтных настроек | hardening, конфигурационные baseline |
| Sensitive Data Exposure | утечка персональных данных | TLS, шифрование, маскирование логов |

## Минимальный процесс защиты

1. На этапе требований фиксируйте security-критерии приемки.
1. В разработке используйте SAST/DAST и dependency scanning.
1. Перед релизом проверяйте критичные сценарии вручную и автоматикой.
1. После релиза мониторьте попытки атак и регулярно обновляйте зависимости.

## Типичные ошибки

- доверие проверкам только на фронтенде;
- отсутствие контроля сторонних библиотек;
- редкие обновления зависимостей и инфраструктурных компонентов.

## Чек-лист качества

- Для API есть единая политика валидации входных данных.
- Проверки авторизации выполняются на сервере для каждого ресурса.
- Security-тесты включены в release pipeline.

## Практическое дополнение

1. Перед применением подхода зафиксируйте измеримую цель (SLA, lead time, error rate, time-to-market).
1. На архитектурном review проверяйте соответствие решения ограничениям безопасности, интеграций и эксплуатации.
1. Добавляйте в backlog минимум одну задачу на снижение архитектурного риска для каждой high-impact зоны.
