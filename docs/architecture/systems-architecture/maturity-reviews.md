# Зрелость и арх
итектурные ревью

Архитектурная зрелость помогает понять, насколько системно организация принимает архитектурные решения и управляет техническими рисками.

## Уровни сложности

### Базовый уровень

- понимать уровни зрелости архитектуры;
- проводить базовую self-assessment;
- выявлять ключевые gaps.

### Средний уровень

- использовать [CMMI](../../glossary.md#abbr-018)/TOGAF maturity ориентиры;
- проводить архитектурные ревью (ATAM/ARC-lite);
- формировать roadmap улучшений.

### Продвинутый уровень

- внедрять регулярный governance цикл;
- связывать зрелость с бизнес-[KPI](../../glossary.md#abbr-047) и рисками;
- масштабировать ревью на portfolio/программы.

## Модели зрелости (кратко)

| Модель | Что оценивает | Где полезна |
| --- | --- | --- |
| CMMI | зрелость процессов разработки | delivery predictability |
| TOGAF maturity | зрелость EA-практики | enterprise architecture governance |
| ACMM/аналогичные | capability архитектурной функции | roadmap эволюции архитектуры |

## Как проводить архитектурное ревью

### ATAM (Architecture Tradeoff Analysis Method)

1. Подготовить архитектурные драйверы ([NFR](../../glossary.md#abbr-053) + риски).
2. Зафиксировать архитектурные подходы и ключевые решения.
3. Проанализировать trade-off сценарии.
4. Сформировать риск/чувствительность/компромиссы.
5. Подготовить рекомендации и roadmap.

### ARC / lightweight архитектурный аудит

- проверка соответствия решения требованиям;
- проверка reliability/security/performance рисков;
- формирование actions с owner и сроком.

## Шаблон отчета ревью

| Раздел | Содержание |
| --- | --- |
| Scope | какие системы и сценарии оценивались |
| Findings | выявленные риски и ограничения |
| Trade-offs | ключевые архитектурные компромиссы |
| Recommendations | приоритетные улучшения |
| Owners/ETA | ответственные и сроки |

## Плюсы и минусы формального ревью

| Плюсы | Минусы |
| --- | --- |
| раннее обнаружение архитектурных рисков | требует времени и вовлечения экспертов |
| прозрачные решения и ответственность | может стать формальностью без follow-up |
| улучшение межкомандной согласованности | нужен стабильный процесс и owner |

## Практические рекомендации

1. Проводите ревью по этапам: inception, pre-release, post-incident.
2. Не делайте ревью только ради compliance.
3. Привязывайте findings к бизнес-рискам и метрикам.
4. Делайте обязательный follow-up по рекомендациям.
## Стандарты и источники

- ATAM method (SEI/CMU references).
- TOGAF maturity guidance.
- CMMI resources.
