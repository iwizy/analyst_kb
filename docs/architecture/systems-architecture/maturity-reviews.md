# Зрелость и архитектурные ревью

Страница задает практический процесс, как оценивать зрелость архитектурной практики, проводить ревью решений и превращать findings в управляемые улучшения.

## Уровни сложности

### Базовый уровень

- понимать, зачем нужна архитектурная зрелость;
- запускать самооценку по ключевым измерениям;
- фиксировать риски и первые улучшения.

### Средний уровень

- проводить формальные ревью по [ATAM](../../glossary.md#abbr-108) или lightweight-аудиту;
- связывать findings с [NFR](../../glossary.md#abbr-053), [SLO](../../glossary.md#abbr-081) и delivery-метриками;
- управлять реестром решений и архитектурным долгом.

### Продвинутый уровень

- строить регулярный governance-цикл на уровне нескольких команд;
- вводить архитектурные quality gates в [CI/CD](../../glossary.md#abbr-017);
- оценивать эффект архитектуры через бизнес-[KPI](../../glossary.md#abbr-047).

## Когда нужен этот процесс

| Сигнал | Что обычно происходит без процесса |
| --- | --- |
| Частые архитектурные споры | решения принимаются ситуативно и повторно пересматриваются |
| Рост инцидентов после релизов | неуправляемые trade-off между скоростью и надежностью |
| Миграции «застревают» | нет четкого риск-реестра и критериев перехода |
| Разные команды принимают несовместимые решения | растет интеграционная сложность и стоимость изменений |

## Модель зрелости архитектурной функции

| Уровень | Характеристика | Ключевой результат |
| --- | --- | --- |
| L1 Initial | решения реактивные, зависят от конкретных людей | локальные улучшения без повторяемости |
| L2 Managed | есть базовые [ADR](../../glossary.md#abbr-003), фиксируются ограничения | появляется прозрачность решений |
| L3 Defined | стандартизированы шаблоны, есть регулярные ревью | решения сопоставимы между командами |
| L4 Quantified | применяются метрики качества архитектуры | trade-off управляются численно |
| L5 Optimizing | непрерывное улучшение и platform governance | устойчивый рост скорости и надежности |

## Измерения зрелости и шкала оценки

Оценка выполняется по шкале `0..4` для каждого измерения.

| Измерение | 0 | 2 | 4 |
| --- | --- | --- | --- |
| Управление решениями | решения не документируются | решения документируются частично | все значимые развилки имеют [ADR](../../glossary.md#abbr-003) и статус |
| Quality attributes | [NFR](../../glossary.md#abbr-053) не формализованы | NFR есть для критичных сервисов | NFR связаны с SLI/SLO и проверяются |
| Архитектурные ревью | проводятся только при кризисе | проводятся перед крупными релизами | встроены в жизненный цикл изменений |
| Observability | логи и метрики фрагментарны | есть базовые дашборды | полная трассировка + error budget контроль |
| Security by design | безопасность после разработки | threat modeling для крупных изменений | security требования встроены в design/review |
| Platform readiness | инфраструктура ручная | частичная автоматизация | platform engineering и стандартизированные guardrails |

Итоговый индекс зрелости: `сумма баллов / число измерений`.

## Форматы архитектурного ревью

| Формат | Когда применять | Длительность | Выход |
| --- | --- | --- | --- |
| [ATAM](../../glossary.md#abbr-108) | крупная развилка или сложная миграция | 1-3 дня | список trade-off, рисков, sensitivity points |
| Lightweight аудит (ARC-lite) | регулярный контроль релизов | 60-120 минут | перечень корректирующих действий |
| Post-incident архитектурный разбор | после значимого сбоя | 60-90 минут | backlog надежности и обновление [ADR](../../glossary.md#abbr-003) |

## Процесс ревью от заявки до закрытия

```kroki-plantuml
@startuml
start
:Intake ревью и определение scope;
:Сбор входов (ADR, C4, NFR, инциденты, метрики);
:Сессия ревью и фиксация findings;
:Оценка риска и приоритизация действий;
:Создание задач и назначение owners;
:Контроль выполнения и проверка эффекта;
stop
@enduml
```

## Входы ревью

| Категория | Минимальный набор |
| --- | --- |
| Контекст | бизнес-цель, границы системы, текущие ограничения |
| Архитектура | актуальные [C4](../../glossary.md#abbr-011)/[UML](../../glossary.md#abbr-092) диаграммы, [ADR](../../glossary.md#abbr-003) |
| Требования | функциональные сценарии + [NFR](../../glossary.md#abbr-053) |
| Эксплуатация | SLI/SLO, инциденты, latency/error trends |
| Риски | открытые риски, зависимости, внешний контур |

## Матрица приоритизации рекомендаций

| Приоритет | Условие | Ожидаемая реакция |
| --- | --- | --- |
| P1 | высокий риск + высокая вероятность | запуск задачи в текущем релизе |
| P2 | высокий риск + средняя вероятность | план в ближайший квартал |
| P3 | средний риск или локальная оптимизация | плановый backlog улучшений |

## Метрики эффективности архитектурного governance

| Метрика | Что показывает | Целевая динамика |
| --- | --- | --- |
| % значимых решений с [ADR](../../glossary.md#abbr-003) | прозрачность архитектурных выборов | рост до стабильного покрытия |
| Findings closure rate | скорость закрытия архитектурных рисков | увеличение доли закрытых действий |
| Incident recurrence | повторяемость инцидентов одного класса | устойчивое снижение |
| Lead time на архитектурные изменения | оперативность изменения платформы | снижение без роста дефектов |
| Change failure rate | влияние архитектурных изменений на стабильность | снижение |

## Типовые ошибки процесса

| Ошибка | Симптом | Как исправить |
| --- | --- | --- |
| Ревью проводится «для галочки» | findings не превращаются в задачи | обязательная связка findings -> backlog -> owner |
| Нет связи с требованиями | решения не защищают ключевые NFR | ввести входной критерий: NFR обязательны |
| Нет проверки эффекта | рекомендации закрыты формально | фиксировать метрику успеха для каждого действия |
| Слишком широкий scope ревью | бесконечные встречи без вывода | ограничивать scope одной развилкой/релизом |
