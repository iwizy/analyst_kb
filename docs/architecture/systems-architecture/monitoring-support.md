# Мониторинг и поддержка

Наблюдаемость (logs, metrics, traces) и процессы поддержки определяют
способность системы быстро обнаруживать и устранять инциденты.

## Уровни сложности

### Базовый уровень

- понимать триаду logs/metrics/traces;
- определять базовые [SLI](../../glossary.md#abbr-080)/[SLO](../../glossary.md#abbr-081);
- поддерживать runbook для типовых инцидентов.

### Средний уровень

- настраивать алерты без шумовых срабатываний;
- проводить postmortem и follow-up;
- связывать наблюдаемость с архитектурными решениями.

### Продвинутый уровень

- строить observability platform;
- внедрять AIOps/корреляцию инцидентов;
- управлять надежностью через error budget практики.

## Минимальный стек наблюдаемости

- метрики сервиса и инфраструктуры;
- структурированные логи;
- distributed tracing;
- алерты по SLI/SLO.

## Operational flow

```kroki-plantuml
@startuml
start
:Сработал алерт;
:Дежурный анализирует метрики/логи/трейсы;
if (Причина ясна?) then (Да)
  :Устранить и проверить восстановление;
else (Нет)
  :Эскалация и war-room;
endif
:Postmortem и действия по предотвращению;
stop
@enduml
```

## Чек-лист качества

- Для критичных сервисов определены [SLO](../../glossary.md#abbr-081)/[SLI](../../glossary.md#abbr-080).
- Runbook покрывает частые инциденты.
- Постмортемы превращаются в план улучшений.

## Практическое дополнение

1. Перед применением подхода зафиксируйте измеримую цель ([SLA](../../glossary.md#abbr-079), lead time, error rate, time-to-market).
1. На архитектурном review проверяйте соответствие решения ограничениям безопасности, интеграций и эксплуатации.
1. Добавляйте в backlog минимум одну задачу на снижение архитектурного риска для каждой high-impact зоны.
