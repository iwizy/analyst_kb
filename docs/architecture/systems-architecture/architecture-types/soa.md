# SOA

SOA (Service-Oriented Architecture) - enterprise-подход, где сервисы стандартизируются через контракты и централизованный governance.

## Когда выбирать

- интеграция большого числа систем и legacy-контуров;
- необходим централизованный контроль контрактов и политик;
- есть корпоративная архитектурная функция.

## Плюсы

- повторное использование сервисов;
- единые правила интеграции;
- прозрачный каталог сервисных контрактов.

## Минусы

- перегруженный интеграционный слой (ESB) как узкое место;
- медленная эволюция из-за тяжелого governance;
- сложная поддержка совместимости множества потребителей.

## Типичные ошибки

- превращение ESB в единый оркестратор всей бизнес-логики;
- отсутствие версии и жизненного цикла контракта;
- перегруженные «универсальные» сервисы.

## Практика

- контракт-first и строгая политика версии;
- SLA и observability по каждому сервису;
- каталог зависимостей «поставщик -> потребители».
