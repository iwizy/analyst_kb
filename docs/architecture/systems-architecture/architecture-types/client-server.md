# Клиент-серверная архитектура

Клиент-серверная модель разделяет систему на клиентский слой (UI/приложение) и серверный слой (бизнес-логика, доступ к данным).

## Когда выбирать

- классические web/mobile системы;
- централизованное хранение данных и политики безопасности;
- понятный путь эволюции от MVP к более сложной архитектуре.

## Плюсы

- простая модель взаимодействия;
- централизованный контроль доступа и логики;
- удобная эксплуатация на ранних этапах.

## Минусы

- сервер становится bottleneck при росте нагрузки;
- ограниченная автономия клиентских команд;
- риск перегрузки API-слоя «всеми задачами сразу».

## Базовая схема

```kroki-plantuml
@startuml
left to right direction
actor "Клиент" as C
rectangle "API Server" as S
database "DB" as D
C --> S : HTTP/gRPC
S --> D : read/write
S --> C : response
@enduml
```

## Практика

- разделяйте API по bounded context;
- добавляйте caching и read-replica на горячие сценарии;
- контролируйте latency, error rate, saturation на сервере.
