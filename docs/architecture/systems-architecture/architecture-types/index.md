# Виды архитектур

Выбор стиля архитектуры - это выбор компромиссов: скорость изменений, сложность эксплуатации, стоимость масштабирования и устойчивость к отказам.

## Сравнение архитектурных стилей

| Стиль | Сильные стороны | Ограничения | Когда подходит |
| --- | --- | --- | --- |
| [Монолит](monolith.md) | быстрый старт, простая эксплуатация | сложнее масштабировать частями | MVP, небольшие команды |
| [Модульный монолит](modular-monolith.md) | управляемые границы без distributed complexity | нужен архитектурный контроль модулей | рост продукта при 1-3 командах |
| [Микросервисы](microservices.md) | независимые релизы, изоляция доменов | высокая операционная сложность | крупные продукты и много команд |
| [Многослойная](multilayer.md) | ясное разделение ответственности | риск «анемичной» доменной модели | корпоративные back-office системы |
| [SOA](soa.md) | сервисная стандартизация на уровне enterprise | тяжёлый governance, ESB-bottleneck | интеграция множества legacy-систем |
| [Микрофронтенды](microfrontends.md) | независимые релизы UI-команд | риск фрагментации UX | большие digital-платформы |
| [Peer-to-peer](p2p.md) | децентрализация и устойчивость к центру | сложная консистентность и безопасность | распределенные сети, edge-сценарии |
| [Клиент-сервер](client-server.md) | понятная базовая модель | центральные узкие места | классические web/mobile системы |

## Как выбрать стиль

1. Определите масштаб продукта и число автономных команд.
1. Зафиксируйте NFR: latency, availability, throughput, RTO/RPO.
1. Оцените зрелость DevOps/SRE и observability.
1. Выберите минимально сложную архитектуру, закрывающую риски.

## Типичные ошибки

- начинать с микросервисов без операционной зрелости;
- держать монолит без модульных границ при росте домена;
- переносить стиль из другого проекта без учета контекста.
