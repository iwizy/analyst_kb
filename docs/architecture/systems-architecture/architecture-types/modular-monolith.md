# Модульный монолит

Модульный монолит сочетает единый деплой и строгие границы модулей. Это частый целевой стиль для зрелого продукта до перехода на микросервисы.

## Когда выбирать

- домен растет, но распределенная архитектура преждевременна;
- нужен контроль зависимостей между бизнес-областями;
- команды готовы к дисциплине архитектурных контрактов.

## Плюсы

- меньше операционной сложности, чем в микросервисах;
- высокая скорость локальной разработки;
- контролируемый путь эволюции к сервисной архитектуре.

## Минусы

- общий релизный контур сохраняется;
- нужна жесткая инженерная дисциплина модульных границ;
- при плохих границах быстро превращается в обычный монолит.

## Практические правила

- `module API first`: доступ только через публичные интерфейсы;
- запрет «сквозных» зависимостей между внутренностями модулей;
- отдельный ownership каждого модуля.

## Типичные ошибки

- «модули» только в названии пакетов;
- общие util-библиотеки как скрытая связность;
- отсутствие метрик межмодульной связности.
