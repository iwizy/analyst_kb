# Микрофронтенды

Микрофронтенды разделяют UI на автономные продуктовые вертикали, которые могут выпускаться разными командами.

## Когда выбирать

- большой frontend и множество независимых доменных команд;
- разный темп изменений в продуктовых областях;
- нужен отдельный ownership по частям интерфейса.

## Плюсы

- независимые релизы UI-частей;
- локальная скорость команд;
- гибкая эволюция интерфейсных модулей.

## Минусы

- риск несогласованного UX;
- дублирование зависимостей и рост bundle size;
- сложнее сквозная аналитика и трассировка ошибок.

## Типичные ошибки

- отсутствие общей design system;
- общий state без четких правил владения;
- разное качество accessibility и performance в модулях.

## Практика

- единые UI-токены, гайдлайн и контракт маршрутизации;
- централизованные web-vitals и error budgets;
- договоренности по межмодульной коммуникации.
