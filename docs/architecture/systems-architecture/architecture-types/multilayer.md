# Многослойная архитектура

Многослойная архитектура разделяет решение по ответственности: presentation, application, domain, infrastructure.

## Когда выбирать

- нужны понятные инженерные границы в enterprise-системе;
- важна управляемая эволюция кода;
- команда развивает backend с выраженной бизнес-логикой.

## Плюсы

- понятное распределение ответственности;
- проще onboard новых разработчиков;
- выше тестируемость при правильных границах.

## Минусы

- риск «толстых сервисов» и аномальной бизнес-логики в контроллерах;
- избыточная многослойность для простых систем;
- overhead на навигацию по коду.

## Типичные ошибки

- инфраструктура проникает в доменный слой;
- отсутствие use-case уровня;
- слойность ради слойности без практической ценности.

## Практика

- зависимость направлена внутрь домена;
- бизнес-правила не зависят от БД/HTTP;
- каждый слой имеет четкий контракт.
