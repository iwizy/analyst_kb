# Планирование и управление временем

> Глоссарий терминов и сокращений: [Открыть](../glossary.md)


Тайм-менеджмент аналитика это управление фокусом, контекст-переключениями и приоритетами. Цель не занять весь календарь, а стабильно поставлять полезный результат.

## Уровни сложности

### Базовый уровень

- применять ежедневное и недельное планирование;
- отличать срочные задачи от важных;
- резервировать время на deep work.

### Средний уровень

- управлять capacity команды и зависимостями;
- использовать velocity, burn-down/burn-up;
- поддерживать баланс между встречами и аналитической работой.

### Продвинутый уровень

- строить систему time governance для команды;
- управлять буферами и риском перегрузки;
- внедрять практики профилактики выгорания.

## Личные техники аналитика

| Техника | Как применять |
| --- | --- |
| Pomodoro | 25/5 или 50/10 для фокус-блоков |
| Time Blocking | заранее резервировать блоки под анализ, встречи, review |
| Eisenhower Matrix | разделять задачи на важные/срочные |
| GTD | выгрузка задач во внешнюю систему + next actions |
| MoSCoW | приоритизация задач по ценности |
| 80/20 | фокус на 20% задач с максимальным эффектом |

## Пример недельного плана

| День | Deep work | Встречи | Результат дня |
| --- | --- | --- | --- |
| Пн | 2 блока по 90 мин (требования) | Planning + sync | обновленный backlog |
| Вт | 1 блок (моделирование) | stakeholder interviews | draft спецификации |
| Ср | 2 блока (аналитика данных) | refinement | решения по приоритетам |
| Чт | 1 блок (документация) | review/demo | согласованные AC |
| Пт | 1 блок (ретро/улучшения) | weekly review | план следующей недели |

## Командное планирование времени

### Capacity planning

- учитывать отпуска, поддержку, обязательные встречи;
- планировать не 100%, а 70-85% емкости;
- оставлять buffer/slack на неопределенность.

### Velocity и story points

- velocity = средний объем завершенных points за N спринтов;
- story points — относительная сложность, не часы;
- ideal days — инженерный ориентир, не календарное обязательство.

### Burn-down / Burn-up

- burn-down показывает, сколько осталось;
- burn-up показывает, сколько сделано и как меняется scope;
- оба графика полезны для раннего обнаружения отклонений.

## Управление зависимостями и буфером

- фиксировать внешние зависимости до sprint planning;
- добавлять slack time на интеграционные риски;
- визуализировать blockers и aging задач.

## Профилактика выгорания

- time-boxing встреч;
- ограничение параллельных задач;
- делегирование и парная работа;
- регулярные паузы и восстановление;
- защита deep-work окон в календаре.

Инструменты:

- календарь + color coding;
- Jira board/filters;
- Trello/Notion для личного контекста;
- focus apps (Pomodoro timers).

## Шаблоны

### Ежедневный план

| Поле | Пример |
| --- | --- |
| 3 главных результата дня | согласовать AC, обновить RTM, подготовить demo |
| Deep work блоки | 10:00-11:30, 14:00-15:00 |
| Встречи | 11:30 refinement, 16:00 sync |
| Риски дня | внешний API response неизвестен |

### Спринт-шаблон по времени

| Элемент | Значение |
| --- | --- |
| Team capacity | 120 ч |
| Планируемый буфер | 20 ч |
| Delivery capacity | 100 ч |
| Planned scope | 30 SP |

## Типичные ошибки

- планирование без буфера;
- календарь, полностью занятый встречами;
- подмена приоритетов срочностью;
- отсутствие регулярного weekly review.## Книги и источники
