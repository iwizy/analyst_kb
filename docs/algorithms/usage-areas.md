# Области использования алгоритмов и структур данных

Алгоритмы и структуры данных применяются в большинстве прикладных задач:
поиск, рекомендации, маршрутизация, анализ событий, риск-модели. Цель аналитика

- выбрать подход, который дает нужный эффект при реалистичных затратах.

## Карта применения

| Сценарий | Что обычно используют | Почему |
| --- | --- | --- |
| Поиск по каталогу | хеш-таблицы, индексы, trie | быстрый доступ и фильтрация |
| Лента событий/очередь | queue, priority queue, heap | контроль порядка обработки |
| Навигация и логистика | графы + Dijkstra/A\* | поиск оптимального маршрута |
| Рекомендации | сортировка, топ-k, графы связности | ранжирование и персонализация |
| Антифрод | деревья решений, графовый анализ | выявление аномалий и связей |
| ETL/обработка данных | циклы, map/reduce, сортировки | трансформация больших наборов |
| Планирование релизов | greedy/приоритизация, dependency graph | баланс ценности и ограничений |

## Пример: маршрутизация заявок в поддержку

Вход: поток заявок с приоритетами и SLA.

Подход:

- `priority queue` для выбора следующей заявки;
- граф зависимостей, если есть блокировки между задачами;
- сортировка/группировка для отчетности по очередям.

Результат:

- предсказуемое время обработки;
- прозрачные правила эскалации;
- снижение SLA-нарушений при пиках нагрузки.

## Пример: обработка транзакций

1. Валидация входа линейным проходом (`O(n)`).
1. Индексирование по ключам в `dict`/`set` для быстрых проверок (`O(1)` в среднем).
1. Сортировка для формирования отчетных витрин (`O(n log n)`).

## Как выбирать алгоритм под задачу

- Определите тип операции: чтение, запись, поиск, ранжирование, обход.
- Зафиксируйте ограничения: объем, задержка, память, консистентность.
- Сравните 2-3 альтернативы по стоимости владения.
- Учитывайте не только скорость, но и объяснимость решения.

## Типичные ошибки

- применение сложного алгоритма без подтвержденной необходимости;
- игнорирование характеристик данных (распределение, дубликаты, skew);
- оптимизация локального шага вместо сквозного процесса.

## Чек-лист качества

- Алгоритм соответствует бизнес-ограничениям и SLA.
- Есть план деградации при росте нагрузки.
- Решение можно поддерживать командой без узкой экспертизы.
