# Структурирование требований

Структурирование превращает набор разрозненных требований в управляемую систему. Без структуры требования быстро теряют связность, дублируются и становятся дорогими в сопровождении.

## Цель

- разделить требования по уровням и пакетам;
- обеспечить единый формат и правила именования;
- упростить трассировку, review и change management.

## Базовые принципы

- один смысл на одно требование;
- единый шаблон для одного типа требований;
- явная иерархия от цели к реализации;
- идентификатор и владелец у каждого требования;
- обязательные связи с источником и проверкой.

## Рабочая модель структуры

1. Бизнес-цели и KPI.
1. Бизнес-требования.
1. Пользовательские сценарии.
1. Системные требования (функциональные и нефункциональные).
1. Критерии приемки и тестовые сценарии.

## Рекомендуемая группировка

Группируйте требования одновременно по двум осям:

- по доменам или бизнес-возможностям;
- по релизам или инкрементам.

Пример:

- Домен "Возвраты": регистрация возврата, расчет компенсации, уведомления.
- Релиз R1: базовый возврат.
- Релиз R2: частичный возврат и автоматические правила.

## Идентификация и атрибуты

Минимальные атрибуты требования:

- ID (например, `FR-RET-023`);
- уровень (бизнес/пользовательское/системное);
- тип (FR/NFR);
- владелец;
- источник;
- приоритет;
- статус;
- ссылка на критерии приемки.

## Пример структуры в репозитории знаний

```text
REQ-
  BR-
  UR-
  SR-
    FR-
    NFR-
  AC-
  TR-
```

Где:

- `BR` - business requirements;
- `UR` - user requirements;
- `SR` - system requirements;
- `AC` - acceptance criteria;
- `TR` - trace links.

## Контроль целостности структуры

Проверяйте регулярно:

- есть ли дубли формулировок;
- есть ли требования без владельца;
- есть ли требования без статуса;
- есть ли "висячие" требования без связи с целями или тестами;
- есть ли противоречия между релизами.

## Типичные ошибки

- смешение разных уровней в одном документе;
- отсутствие правил именования ID;
- хаотичная группировка по папкам или таблицам;
- хранение требований в нескольких несинхронизированных источниках.

## Практические советы

- зафиксировать единый словарь терминов и использовать его во всех артефактах;
- договориться о статусной модели требований (Draft, Reviewed, Approved, Deprecated);
- проводить структурный review перед каждой крупной итерацией;
- автоматически искать дубли и orphan-требования.

## Чек-лист качества

- требование можно найти по ID за 30 секунд;
- понятно, к какому релизу и домену относится требование;
- для критичных требований есть связи с тестами;
- структура масштабируется без ручного переписывания всего массива требований.
