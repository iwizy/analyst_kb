# Уровни требований

Уровни требований показывают связь между бизнес-целями и реализацией. Разделение по уровням помогает не смешивать стратегические ожидания, пользовательские сценарии и технические решения.

## Зачем разделять требования по уровням

- сохраняется причинно-следственная связь от цели к функциональности;
- проще согласовывать объем со стейкхолдерами;
- повышается трассируемость требований;
- меньше риск реализовать технически верное, но бизнес-бесполезное решение.

## Основные уровни

### Бизнес-требования

Описывают, какую ценность проект должен дать бизнесу и по каким метрикам будет виден результат.

- Фокус: цели, KPI, ограничения, критерии успеха.
- Вопрос: "Зачем мы делаем систему?"
- Подробнее: [Бизнес-требования](business.md)

### Пользовательские требования

Описывают, какие задачи пользователь должен решать с помощью системы.

- Фокус: сценарии использования, роли, пользовательские ожидания.
- Вопрос: "Что пользователь должен уметь сделать?"
- Подробнее: [Пользовательские требования](user.md)

### Системные требования

Описывают, как система должна быть устроена и как она должна работать, чтобы реализовать пользовательские и бизнес-цели.

- Фокус: функции, ограничения, интерфейсы, характеристики качества.
- Вопрос: "Какими средствами система обеспечит нужный результат?"
- Подробнее: [Системные требования](system.md)

## Связь уровней

1. Бизнес-требования задают целевую ценность.
2. Пользовательские требования переводят ценность в сценарии.
3. Системные требования переводят сценарии в реализацию и проверку.

Если связь между уровнями потеряна, команда получает высокий риск лишней функциональности, конфликтов при приемке и перерасхода ресурсов.

## Итог

Работа по уровням это основа управляемых требований: она делает решения прозрачными и позволяет объяснить каждую реализуемую функцию через бизнес-цель.
