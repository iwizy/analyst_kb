# Пользовательские требования

Пользовательские требования описывают, какие задачи и в каком контексте пользователь должен решать с помощью системы. Они связывают бизнес-цели и инженерную реализацию.

## Цель

- описать ожидаемое поведение системы с точки зрения роли пользователя;
- зафиксировать критерии успешного выполнения сценария;
- подготовить основу для системных требований и тестов.

## Что фиксировать

- роль пользователя;
- контекст и триггер действия;
- ожидаемый результат для пользователя и бизнеса;
- ограничения и граничные случаи;
- критерии приемки.

## Форматы описания

- User Story;
- Use Case;
- сценарии/потоки (happy path + edge-cases).

Подбор формата зависит от сложности:

- простой поток в спринте: User Story + acceptance criteria;
- сложный процесс с альтернативами: Use Case;
- критичные интеграционные сценарии: сценарная карта + ограничения.

## Процесс подготовки

1. Собрать пользовательские задачи через интервью и наблюдение.
1. Уточнить цели ролей и частоту сценариев.
1. Зафиксировать основной и альтернативные потоки.
1. Добавить измеримые критерии приемки.
1. Согласовать с бизнесом, разработкой и QA.

## Пример

```text
UR-041
Как оператор поддержки,
я хочу видеть историю обращений клиента за 12 месяцев,
чтобы быстрее решать повторные инциденты.

Acceptance Criteria:
- отображаются обращения за 12 месяцев;
- доступна фильтрация по статусу и приоритету;
- экран открывается не более чем за 2 секунды при стандартной нагрузке.
```

## Частые ошибки

- "хочу кнопку" вместо описания задачи пользователя;
- отсутствие критериев приемки;
- объединение нескольких независимых сценариев в одну историю;
- отсутствие связи с бизнес-целью или KPI.

## Практические рекомендации

- писать требования от задачи роли, а не от интерфейсного элемента;
- выделять отдельные требования для исключений и edge-cases;
- проверять, что критерии приемки могут быть протестированы;
- после согласования связывать пользовательские требования с системными.

## Чек-лист качества

- понятна роль и контекст использования;
- описан измеримый результат;
- есть критерии приемки;
- указан приоритет;
- есть связь с бизнес-требованием.
