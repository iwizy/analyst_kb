# Пользовательские требования

Пользовательские требования описывают, какие задачи и в каком контексте пользователь должен выполнять с помощью системы. Они соединяют бизнес-цели и будущую функциональность продукта.

## Что фиксируют пользовательские требования

- роли пользователей и их цели;
- сценарии использования;
- ожидания по результату действий;
- ключевые условия и ограничения сценария;
- критерии успешного выполнения задачи.

## Форматы описания

- User Story: `Как <роль>, я хочу <действие>, чтобы <ценность>`.
- Use Case: роль, основной поток, альтернативные потоки, исключения.
- Сценарий процесса: шаги пользователя и реакция системы.

## Процесс разработки пользовательских требований

1. Сбор пользовательских задач через интервью, наблюдение и обратную связь.
2. Кластеризация сценариев по ролям и процессам.
3. Выделение обязательных и опциональных сценариев.
4. Формализация в выбранном формате (story, use case, сценарии).
5. Согласование с бизнесом, разработкой и тестированием.

## Критерии качества

- понятны бизнесу и команде;
- не содержат технической реализации;
- имеют проверяемые критерии приемки;
- связаны с конкретной бизнес-целью;
- не противоречат другим сценариям.

## Пример

`Как менеджер поддержки, я хочу видеть историю обращений клиента за 12 месяцев, чтобы быстрее решать повторные инциденты.`

Критерии приемки:

- отображаются обращения за последние 12 месяцев;
- доступна фильтрация по статусу и приоритету;
- время открытия истории не более 2 секунд при стандартной нагрузке.

## Частые ошибки

- общие формулировки без ценности для роли;
- отсутствие критериев приемки;
- смешение нескольких сценариев в одном требовании;
- отсутствие связи с приоритетами релиза.

## Итог

Пользовательские требования делают бизнес-цели прикладными: команда понимает, какие сценарии важны для пользователя и как проверить результат.
