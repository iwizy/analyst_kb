# Трассировка требований

Трассировка требований обеспечивает видимость связей между бизнес-целями, требованиями, реализацией и тестами. Она позволяет быстро оценивать влияние изменений и доказывать полноту покрытия.

## Цель

- подтвердить, что нужные требования реализованы и проверены;
- исключить "лишнюю" функциональность без источника ценности;
- ускорить impact analysis при изменениях;
- поддержать аудит и соответствие стандартам.

## Направления трассировки

- Прямая: цель -> требования -> задачи -> тесты.
- Обратная: дефект/код/тест -> требование -> бизнес-цель.
- Двусторонняя: постоянная связность в обоих направлениях.

## Минимальная модель связей

- BR -> UR/SR;
- UR/SR -> задача разработки;
- UR/SR -> тест-кейс;
- CR -> затронутые требования и тесты;
- дефект -> нарушенное требование.

## Процесс внедрения

1. Определить объекты трассировки и владельцев.
1. Ввести единый формат идентификаторов.
1. Создать RTM (или эквивалент в ALM-инструменте).
1. Заполнить связи для критичных требований.
1. Встроить обновление связей в Definition of Done.
1. Проводить регулярную ревизию покрытия перед релизом.

## Пример RTM

| Requirement ID | Business Goal | Dev Task | Test Case | Статус |
| --- | --- | --- | --- | --- |
| BR-012 | Сократить время обработки претензии | DEV-142 | QA-301 | In progress |
| FR-024 | Восстановление пароля | DEV-187 | QA-355 | Passed |
| NFR-021 | p95 <= 300 мс | DEV-211 | PERF-022 | Planned |

## Метрики качества трассировки

- Coverage: доля требований со связями к задачам и тестам.
- Orphan count: количество элементов без связей.
- Impact analysis time: время оценки влияния CR.
- Defect traceability: доля дефектов с привязкой к требованию.
- Актуальность RTM: доля связей, подтвержденных в текущем релизе.

## Типичные проблемы

- трассировка ведется вручную и нерегулярно;
- модель связей чрезмерно сложна;
- отсутствуют единые ID;
- актуализация делается только перед аудитом.

## Практические рекомендации

- начинать с минимальной модели, затем расширять;
- автоматизировать ссылки между требованиями, задачами и тестами;
- закрепить ответственность за качество связей;
- проверять orphan-элементы на каждой релизной ревизии.

## Чек-лист перед релизом

- у требований из релизного scope есть тестовое покрытие;
- нет задач без исходного требования;
- все CR связаны с затронутыми артефактами;
- критичные дефекты имеют обратную связь на требования.
