# Управление изменениями требований

Изменения требований неизбежны. Задача change management сохранить гибкость продукта без потери предсказуемости по срокам, бюджету и качеству.

## Цель

- обеспечить управляемый жизненный цикл изменений требований;
- минимизировать негативное влияние изменений на сроки, бюджет и качество;
- сохранить согласованность baseline, трассировки и релизного плана.

## Когда изменение проходит через Change Request

CR обязателен, если изменение:

- влияет на релизный scope;
- меняет критерии приемки;
- затрагивает архитектуру, интеграции, безопасность, регуляторику;
- влияет на сроки, бюджет, состав команды;
- меняет внешние обязательства (контракт/SLA).

## Роли

| Роль | Ответственность |
| --- | --- |
| Инициатор | Формулирует изменение и бизнес-причину |
| BA | Оформляет CR, координирует impact analysis |
| PO/Заказчик | Подтверждает ценность и приоритет |
| Tech Lead/Архитектор | Оценивает техническое влияние и риски |
| QA | Оценивает влияние на тесты и регрессию |
| CCB/Руководитель направления | Принимает финальное решение для major-изменений |

## Workflow по типам CR

### Minor Change

Когда использовать: локальные изменения без значимого влияния на сроки и архитектуру.

Поток:

1. Регистрация CR.
1. Быстрый impact analysis (BA + Tech Lead + QA).
1. Решение PO.
1. Планирование в ближайший спринт/релиз.
1. Реализация и закрытие CR.

### Major Change

Когда использовать: изменение scope/архитектуры/комплаенса, риск сдвига релиза.

Поток:

1. Регистрация CR.
1. Полный impact analysis (scope, сроки, бюджет, риски, операционное влияние).
1. Рассмотрение на CCB или product governance board.
1. Решение: Approved / Rejected / Deferred.
1. Перепланирование roadmap и релиза.
1. Реализация, верификация, обновление baseline.

## CCB vs продуктовая команда

| Контекст | Кто принимает решение |
| --- | --- |
| Minor изменение внутри релиза | PO + delivery команда |
| Cross-team изменение или риск релиза | CCB / governance board |
| Регуляторное изменение | CCB с участием compliance/legal |
| Архитектурное изменение платформенного уровня | Архкомитет + CCB |

## Жизненный цикл Change Request

1. Регистрация.
1. Классификация (minor/major, тип изменения).
1. Impact analysis.
1. Решение и согласование.
1. План внедрения.
1. Реализация и проверка.
1. Закрытие и обновление baseline.

## Шаблоны документов

### Change Request Form

| Поле | Описание |
| --- | --- |
| CR ID | Уникальный идентификатор |
| Инициатор | Кто подал изменение |
| Тип | Minor / Major |
| Причина | Бизнес, дефект, регуляторика, техдолг |
| Описание изменения | Что именно меняется |
| Срочность | До какого срока требуется решение |
| Затронутые требования | Ссылки на BR/UR/SR/NFR |

### Impact Assessment

| Область | Вопрос | Оценка |
| --- | --- | --- |
| Scope | Что добавляется/удаляется/меняется | ... |
| Schedule | Как меняется критический путь | ... |
| Cost | Дополнительные трудозатраты и бюджет | ... |
| Architecture | Влияние на API, данные, интеграции | ... |
| QA | Объем регрессии, новые тесты | ... |

## Связь CR с трассировкой и актуальностью

Обязательные действия при Approved CR:

1. Обновить затронутые требования и их версии.
1. Обновить RTM связи `требование -> задача -> тест`.
1. Выполнить переприоритизацию backlog и пересмотреть релизный срез.
1. Зафиксировать изменения в decision log.
1. Проверить актуальность связанных документов.

## Пример

CR-27: добавить статус «Частично оплачен».

Итог анализа:

- затронуты API статусов и биллинговая логика;
- нужно обновить 18 регрессионных тестов;
- влияние на релиз: +3 рабочих дня;
- решение: Approved для следующего инкремента с корректировкой roadmap.

## Настройка инструментов

### Jira

- отдельный issue type `Change Request`;
- обязательные поля: тип CR, причина, impact, решение;
- workflow: Draft -> Analysis -> Review -> Approved/Rejected/Deferred -> Implemented -> Closed;
- автоматические проверки links к требованиям и тестам.

### ServiceNow

- использовать change templates по категориям (minor/major);
- настроить CAB/CCB согласование для major изменений;
- связать change records с релизами и CMDB при необходимости.

### ALM-модули (Polarion/Jama/DOORS)

- baseline before/after CR;
- trace impact report;
- контроль полноты approval history.

## Типичные ошибки

- устные изменения без регистрации CR;
- impact analysis только силами разработки;
- отсутствие связи CR с трассировкой;
- закрытие CR без обновления baseline.

## Практические рекомендации

- использовать единый шаблон CR для всех команд;
- установить SLA обработки CR по критичности;
- разделять fast-track для minor и governance-процесс для major;
- регулярно анализировать долю изменений «в обход» процесса.

## Чек-лист закрытия Change Request

- решение по CR зафиксировано и согласовано;
- обновлены затронутые требования и их версии;
- RTM и тестовое покрытие актуализированы;
- выполнена переприоритизация backlog при необходимости;
- baseline и decision log обновлены.

## Метрики

- среднее время обработки CR;
- доля Approved/Rejected/Deferred;
- доля CR с полным impact analysis;
- доля CR, повлиявших на перепланирование релиза;
- доля срочных изменений вне утвержденного workflow.
