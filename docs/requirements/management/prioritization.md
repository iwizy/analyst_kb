# Приоритезация требований

Приоритизация требований определяет порядок реализации, когда ресурсы ограничены. Ее задача - максимизировать ценность релиза при допустимом уровне риска и затрат.

## Цель приоритизации

- Сфокусировать команду на требованиях с наибольшим вкладом в бизнес-результат.
- Снизить риск срыва релиза из-за перегруженного объема.
- Сделать прозрачными решения о том, что входит в релиз, а что переносится.

## Критерии оценки

На практике лучше использовать фиксированный набор критериев:

- бизнес-ценность;
- влияние на пользователя;
- срочность и регуляторные сроки;
- трудоемкость реализации;
- технический риск и архитектурные зависимости;
- стоимость задержки (cost of delay).

## Методы приоритизации и когда их использовать

### MoSCoW

Подходит для быстрой декомпозиции объема релиза.

- Must: без этого релиз не имеет смысла.
- Should: важно, но можно перенести.
- Could: опционально при наличии ресурса.
- Won't (for now): явно отложено.

### RICE

Подходит для продуктовой среды с большим бэклогом, где важна сравнительная оценка.
Формула: `(Reach * Impact * Confidence) / Effort`.

### Weighted Scoring

Подходит для кросс-функциональных команд, когда нужно согласовать несколько критериев с весами.

### Kano

Подходит для балансировки обязательных и "восхищающих" возможностей в пользовательских продуктах.

### Метод 100 баллов

Полезен для воркшопов со стейкхолдерами, когда нужно быстро собрать коллективную оценку.

## Процесс приоритизации

1. Подготовка бэклога
   Требования очищаются от дублей и приводятся к единому формату.
2. Выбор модели оценки
   Команда фиксирует метод и критерии до начала оценки.
3. Оценка требований
   Каждое требование получает значения по критериям, затем рассчитывается приоритет.
4. Проверка зависимостей
   Приоритет корректируется с учетом технических и организационных ограничений.
5. Согласование релизного скоупа
   Финальный список утверждается с прозрачным объяснением компромиссов.
6. Регулярный пересмотр
   Приоритеты обновляются по событию (новые данные, изменения рынка) и по расписанию.

## Артефакты

- Приоритизированный backlog с датой актуализации.
- Журнал решений по спорным требованиям.
- Матрица критериев и весов.
- Релизный скоуп и список отложенных требований.

## Типичные ошибки

- Приоритет задается только "по голосу самого громкого".
- Нет единых критериев и шкал оценки.
- Оценка проводится без технической команды.
- Срочные запросы постоянно обходят процесс.
- Приоритеты не пересматриваются после изменений контекста.

## Признаки здоровой приоритизации

- Для каждого требования есть понятное обоснование приоритета.
- Отложенные требования явно зафиксированы и не теряются.
- Команда может объяснить, какие требования дают основную ценность релиза.
- Изменение приоритета связано с конкретным событием или данными.

## Итог

Приоритизация работает, когда это не разовая встреча, а повторяемый процесс с прозрачными критериями и регулярным пересмотром.
