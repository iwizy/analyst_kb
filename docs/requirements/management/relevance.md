# Поддержка актуальности требований

> Глоссарий терминов и сокращений: [Открыть](../../glossary.md)


Актуальность требований означает, что backlog и спецификации соответствуют текущим бизнес-целям, пользовательским сценариям и техническому контексту. Без регулярной ревизии команда реализует устаревшие или конфликтующие требования.

## Цель

- удерживать требования в рабочем и проверяемом состоянии;
- своевременно убирать дубли, «мертвые» и противоречивые элементы;
- синхронизировать требования, тесты, roadmap и фактическое использование функций.

## Почему требования теряют актуальность

- смена бизнес-приоритетов;
- новая пользовательская аналитика;
- изменения регуляторики и договорных обязательств;
- эволюция архитектуры и интеграций;
- накопление backlog без регулярного cleanup.

## Признаки неактуальности

- нет владельца, статуса или даты последнего review;
- нет связи с текущей целью/инициативой;
- AC не отражают актуальный процесс;
- требование долго не попадает в план без явного решения;
- есть конфликт с новыми требованиями или контрактами API.

## Процесс поддержания актуальности

1. Плановый review по cadence команды.
1. Подтверждение бизнес-ценности и приоритета.
1. Проверка реализуемости и технических допущений.
1. Очистка: объединение дублей, перевод устаревших требований в `Obsolete/Deferred`.
1. Синхронизация: задачи, тесты, документация, roadmap.
1. Фиксация решений в decision log.

## Методики регулярных ревью

### Backlog Grooming / Refinement

- частота: еженедельно или раз в 2 недели;
- цель: поддерживать готовность требований к планированию;
- выход: очищенный и приоритизированный backlog.

### Roadmapping Sessions

- частота: раз в месяц/квартал;
- цель: сверить требования с целями релизов и capacity;
- выход: обновленные релизные срезы.

### Quarterly Business Review (QBR)

- частота: раз в квартал;
- цель: пересмотр крупных инициатив по данным бизнеса;
- выход: решения по продолжению, пересмотру или закрытию направлений.

## Пример agenda для review актуальности

1. Обзор изменений бизнес-целей и KPI.
1. Список требований-кандидатов на Update/Obsolete/Merge.
1. Анализ влияния на roadmap, тесты и интеграции.
1. Решения и владельцы действий.
1. Обновление decision log и статусов.

## Шаблон отчета по устаревшим требованиям

| Поле | Описание |
| --- | --- |
| Requirement ID | Идентификатор требования |
| Последний review | Дата |
| Причина устаревания | Бизнес, техконтекст, регуляторика, дублирование |
| Рекомендация | Keep / Update / Merge / Obsolete |
| Влияние | Scope, сроки, тесты, интеграции |
| Ответственный | Владелец решения |
| Срок действия | До какой даты решение должно быть исполнено |

Пример строки:

| Requirement ID | Последний review | Причина устаревания | Рекомендация | Влияние | Ответственный | Срок действия |
| --- | --- | --- | --- | --- | --- | --- |
| FR-RET-045 | 2025-10-12 | Функция не используется (0.3% трафика) | Obsolete | Удалить 6 тестов, обновить help-page | PO | 2026-03-01 |

## План действий по закрытию устаревших требований

1. Зафиксировать решение и причину.
1. Обновить статус требования (`Obsolete`, `Deferred`, `Merged`).
1. Пересмотреть связанные тесты, задачи, CR, документы.
1. Обновить roadmap и release notes.
1. Проверить, что изменения отражены в трассировке.

## Интеграция с process mining и продуктовой аналитикой

Источники сигналов неактуальности:

- low usage features (продуктовая аналитика);
- циклы процесса с ручными обходами (process mining);
- рост ошибок/эскалаций по конкретным сценариям;
- отклонения SLA от ожиданий.

Практика:

- связывать метрики использования с требованиями;
- помечать «кандидатов на cleanup» в backlog;
- обсуждать кандидатов на QBR и roadmap review.

## Decision Log для отказа от требований

Рекомендуемая структура записи:

| Поле | Пример |
| --- | --- |
| Decision ID | DEC-RET-2026-014 |
| Requirement ID | FR-RET-045 |
| Решение | Obsolete |
| Обоснование | Низкая ценность и дублирование BI-отчетом |
| Последствия | Удаление тестов QA-771..QA-776 |
| Утвердил | PO + Tech Lead |
| Дата пересмотра | 2026-06-01 |

## Метрики актуальности

- доля требований, пересмотренных за период;
- доля требований без владельца;
- средний возраст требования до первого review;
- conflict count: количество конфликтов между требованиями, выявленных за период;
- доля требований в статусе Obsolete/Deferred с корректным обоснованием;
- доля требований с подтвержденной ценностью в текущем roadmap.

## Инструменты

| Инструмент | Для чего использовать |
| --- | --- |
| Jira / Azure DevOps | Статусы требований, backlog cleanup, контроль срока review |
| Confluence | Decision log и протоколы ревью |
| Product analytics (Amplitude/GA) | Сигналы низкого использования функций |
| Process mining (Celonis и аналоги) | Поиск ручных обходов и устаревших этапов процесса |
| BI-дашборды | Мониторинг метрик актуальности и aging требований |

## Чек-лист завершения ревью актуальности

- всем проверенным требованиям обновлены статусы;
- для устаревших требований есть решение и обоснование;
- связанные тесты и задачи пересмотрены;
- roadmap и приоритеты синхронизированы с решениями;
- решения зафиксированы в decision log.

## Типичные ошибки

- ревизия только перед релизом;
- cleanup без синхронизации с тестами и трассировкой;
- отсутствие formal decision log;
- хранение статусов в чатах вместо единой системы.

## Практические рекомендации

- сделать review актуальности частью регулярного процесса;
- назначить владельца для каждого high-impact требования;
- использовать метрики usage и process mining как вход для cleanup;
- не удалять требования без фиксации решения и влияния.
