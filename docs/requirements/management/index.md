# Управление требованиями

Управление требованиями это операционная система для всего жизненного цикла требований: от согласования baseline до контроля изменений, трассировки и актуальности.

## Цель

- удерживать единое понимание требований у всех ролей;
- управлять изменениями без потери предсказуемости;
- обеспечивать связь между целями, реализацией и результатами проверки.

## Операционная модель

### Единый источник правды

Требования, статусы, решения и связи должны храниться в одном управляемом контуре (Jira/ALM + документация).

### Регулярные циклы

| Цикл | Частота | Результат |
| --- | --- | --- |
| Backlog refinement | Еженедельно | Актуальные и готовые требования |
| Review качества | Раз в 1–2 недели | Снижение дефектов формулировок |
| Review актуальности | Ежемесячно | Очистка устаревших элементов |
| Release trace review | Перед релизом | Проверка покрытия и связей |

### Формальные точки контроля

- baseline для high-impact scope;
- impact analysis для значимых изменений;
- quality gate перед статусом Approved;
- post-release review по дефектам требований.

## Сквозной цикл управления

1. Структурировать требования и назначить владельцев.
1. Провести quality review и согласовать baseline.
1. Приоритизировать релизный срез.
1. Связать требования с задачами и тестами.
1. Управлять изменениями через CR.
1. Регулярно пересматривать актуальность и качество.

## Ключевые артефакты

- requirement baseline;
- decision log;
- traceability matrix/дашборд;
- реестр CR;
- отчеты по качеству требований.

## Метрики здоровья процесса

- coverage связей (`requirement -> task -> test`);
- orphan count (несвязанные элементы);
- среднее время обработки CR;
- доля требований, прошедших ревизию за период;
- доля дефектов, вызванных некачественными требованиями.

## Типичные ошибки

- управление требованиями включается только перед релизом;
- изменения проходят «в обход» процесса;
- решения остаются в чатах, не в системе;
- нет владельца качества и актуальности требований.

## Практические рекомендации

- определить минимально достаточный регламент и соблюдать его;
- привязать activities к календарю команды;
- автоматизировать контроль обязательных полей и связей;
- использовать метрики для улучшения процесса, а не только для отчетности.
