# Управление требованиями

Управление требованиями это системный процесс, который удерживает требования согласованными, актуальными и трассируемыми на всем жизненном цикле продукта.

## Цель

- поддерживать единое понимание требований у всех ролей;
- контролировать изменения без потери предсказуемости;
- обеспечивать связность между целями, реализацией и проверкой.

## Операционная модель

### Единый источник правды

Требования, статусы и решения хранятся в одном контуре (backlog/система требований).

### Регулярные циклы

- weekly backlog refinement;
- review актуальности требований по расписанию;
- релизная ревизия трассировки и критериев приемки.

### Формальные точки контроля

- baseline перед разработкой крупного блока;
- impact analysis для значимых изменений;
- quality gate перед переводом в Approved.

## Сквозной цикл управления

1. Подготовить и структурировать требования.
1. Проверить качество и согласовать baseline.
1. Приоритизировать релизный scope.
1. Связать требования с задачами и тестами.
1. Управлять изменениями через CR.
1. Проводить регулярную ревизию актуальности.

## Ключевые артефакты

- requirement baseline;
- decision log;
- traceability matrix;
- change request register;
- метрики качества требований.

## Метрики здоровья процесса

- доля требований с полными связями (coverage);
- доля требований без владельца;
- среднее время обработки CR;
- доля требований, пересмотренных за период;
- количество дефектов, вызванных некачественными требованиями.

## Типичные ошибки

- управление требованиями выполняется только перед релизом;
- изменения вносятся "в обход" формального процесса;
- решения фиксируются в переписках, а не в системе;
- нет регулярной ревизии актуальности.

## Практические рекомендации

- определить минимально достаточный процесс и соблюдать его регулярно;
- привязать активности управления требованиями к календарю команды;
- сделать quality gate обязательным для high-impact требований;
- использовать метрики не для отчетности, а для улучшения процесса.
