# Функциональные требования

Функциональные требования описывают, какие действия система должна поддерживать для пользователей и внешних систем. Это основа для реализации бизнес-логики и построения функциональных тестов.

## Цель

- зафиксировать ожидаемое поведение системы;
- определить условия выполнения и результат действий;
- подготовить проверяемые критерии приемки.

## Что обязательно включать

- ID и название требования;
- источник (бизнес- или пользовательское требование);
- предусловия;
- основной поток;
- альтернативные и ошибочные потоки;
- постусловия;
- критерии приемки.

## Шаблон

```text
FR-XXX
Источник: <UR/BR>
Описание: <что делает система>
Предусловия: <когда сценарий доступен>
Основной поток: <шаги>
Альтернативные потоки: <исключения>
Постусловия: <состояние после выполнения>
Acceptance Criteria: <проверяемые критерии>
```

## Пример

```text
FR-024
Источник: UR-041
Описание: пользователь может восстановить пароль по email.
Предусловия: учетная запись активна.
Основной поток:
1. Пользователь вводит email.
2. Система отправляет одноразовую ссылку.
3. Пользователь задает новый пароль.
Постусловия: пароль обновлен.
Acceptance Criteria:
- ссылка одноразовая и действует 30 минут;
- старый пароль становится недействительным;
- пользователю отправляется уведомление о смене пароля.
```

## Процесс подготовки

1. Взять пользовательский сценарий как источник.
1. Выделить атомарные функции, не смешивая разные цели.
1. Описать основной и альтернативные потоки.
1. Добавить критерии приемки в измеримом виде.
1. Связать требование с тест-кейсами и NFR.

## Типичные ошибки

- несколько независимых функций в одном требовании;
- отсутствие негативных сценариев;
- формулировки без проверяемого результата;
- копирование интерфейсного текста вместо бизнес-логики.

## Практические рекомендации

- использовать глаголы действия (создать, рассчитать, заблокировать, отправить);
- проверять требование вопросом: "можно ли по нему написать тест сразу?";
- отдельно фиксировать ошибки и исключения;
- добавлять ссылку на исходный пользовательский сценарий.

## Чек-лист качества

- требование атомарно;
- описаны входы и выходы;
- есть критерии приемки;
- понятна связь с бизнес-ценностью;
- требование не конфликтует с другими FR.
