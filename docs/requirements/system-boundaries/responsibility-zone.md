# Зона ответственности проекта

Зона ответственности определяет, кто принимает решения и кто отвечает за результат по требованиям на каждом этапе: от сбора до релиза. Явная модель ответственности снижает конфликтность и ускоряет согласования.

## Цель

- назначить владельцев требований и решений;
- зафиксировать decision rights по ключевым активностям;
- определить эскалацию для конфликтных случаев.

## Ключевые роли

- Заказчик/PO: бизнес-ценность, приоритеты, финальные продуктовые решения;
- BA: качество требований, фасилитация, фиксация решений;
- Tech Lead/Архитектор: реализуемость, техограничения, архитектурные компромиссы;
- Разработка: оценка и реализация;
- QA: тестируемость, критерии приемки, проверка соответствия;
- PM/Delivery Manager: сроки, риски, прозрачность статусов;
- Смежные функции: безопасность, legal, operations.

## Практический инструмент: RACI

| Активность | R | A | C | I |
| --- | --- | --- | --- | --- |
| Формулировка BR/UR | BA | PO | QA, Tech Lead | PM |
| Формирование SR/FR/NFR | BA, Tech Lead | Tech Lead | QA, PO | PM |
| Приоритизация backlog | PO, BA | PO | Tech Lead, QA | Команда |
| Согласование baseline | BA | PO | Tech Lead, QA, Legal | PM |
| Обработка CR | BA | PO/CCB | Tech Lead, QA | Команда |
| Приемка результата | QA, PO | PO | BA, Tech Lead | PM |

Легенда: `R` Responsible, `A` Accountable, `C` Consulted, `I` Informed.

## Каталог decision rights

| Тип решения | Кто принимает | Кто консультирует |
| --- | --- | --- |
| Изменение бизнес-приоритета | PO/Заказчик | BA, Tech Lead |
| Изменение архитектурного подхода | Tech Lead/Архитектор | BA, QA, Security |
| Изменение, влияющее на релиз | CCB / PO | PM, BA, Tech Lead |
| Комплаенс-ограничения | Legal/Security owner | BA, PO, Tech Lead |

## Матрица эскалации

| Ситуация | Куда эскалировать | SLA решения |
| --- | --- | --- |
| Конфликт бизнес-ценности и техограничений | PO + Tech Lead | 2 рабочих дня |
| Регуляторный риск | PO + Legal + Security | 1 рабочий день |
| Cross-team блокер | Delivery Manager + руководители команд | 2 рабочих дня |
| Изменение со сдвигом релиза | CCB / governance board | 3 рабочих дня |

## Эскалационный поток

```kroki-plantuml
@startuml
start
:Выявлен конфликт/блокер;
if (Можно решить на рабочей сессии?) then (Да)
  :BA + PO + Tech Lead фиксируют решение;
  :Обновление требований и decision log;
  stop
else (Нет)
  :Эскалация по матрице SLA;
  :Решение PO/CCB/Legal;
  :Обновление baseline и CR;
  stop
endif
@enduml
```

## Процесс настройки ответственности

1. Определить полный список активностей по требованиям.
1. Назначить R/A/C/I для каждой активности.
1. Проверить правило «один A на одну активность».
1. Утвердить матрицу и опубликовать ее в рабочем контуре.
1. Пересматривать матрицу при изменении оргструктуры или процесса.

## Шаблон протокола эскалации

| Поле | Содержание |
| --- | --- |
| Escalation ID | ESC-YYYY-MM-DD-XX |
| Причина | Конфликт приоритета / архитектурный риск / комплаенс |
| Затронутые требования | ID списка |
| Текущий риск | Влияние на scope/срок/качество |
| Решение | Принятое действие |
| Ответственный | Роль/ФИО |
| Срок контроля | Дата подтверждения закрытия |

## Шаблон договоренностей по ответственности

| Поле | Что фиксировать |
| --- | --- |
| Agreement ID | Уникальный идентификатор договоренности |
| Контекст | Инициатива/релиз/процесс |
| Область решения | Какие требования или артефакты покрывает договоренность |
| Decision rights | Кто принимает финальное решение |
| RACI | Распределение ролей по активности |
| SLA решения | Допустимый срок на принятие решения |
| Эскалация | Куда и когда эскалировать конфликт |
| Дата пересмотра | Когда договоренность пересматривается |

Пример: `AGR-RET-02` фиксирует, что по изменениям SLA принимает решение PO (`A`), Tech Lead и QA участвуют как `C`, а спорные случаи эскалируются в CCB в течение 2 рабочих дней.

## Пример

В проекте интеграции платежей финальная ответственность за безопасность требований закреплена за CISO (`A`), Tech Lead и BA участвуют как `C`, разработка как `R` по реализации мер.

## Типичные проблемы

- роль назначена, но границы полномочий размыты;
- несколько ролей одновременно претендуют на `A`;
- решения принимаются вне согласованной матрицы;
- качество требований не закреплено за конкретным владельцем.

## Практические рекомендации

- для каждого high-impact требования указывать owner и approver;
- фиксировать эскалацию в decision log, не в переписке;
- не запускать реализацию без подтвержденного `A`;
- синхронизировать RACI с workflow в Jira/ALM.

## Чек-лист качества

- у каждой активности есть `R` и ровно один `A`;
- правила эскалации известны команде;
- роли ответственности отражены в инструментах;
- изменения ответственности фиксируются формально.
